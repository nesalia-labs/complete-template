## Section C1 — Provisioning Bottleneck Map (4-6 min → 2 min target)

**Author:** C1 specialist. **Scope:** decompose the 12-step provisioning flow, identify parallelization wins, evaluate the 2-min target, and produce a sequence diagram. Re-evaluates feasibility doc Risk #5 (300s ceiling) in light of the 2026-06-15 Vercel 1800s update.

### C1.1 — Risk #5 is obsolete, but C1's floor is unchanged in shape

The feasibility doc's Risk #5 (Vercel Function 300s ceiling) is **obsolete** as of 2026-06-15. The new envelope is:

- **Pro and Enterprise:** 800s default maximum, 1800s extended maximum (30 min) on Node.js 20/22/24 and Python 3.12-3.14 runtimes, **beta**.
- **Opt-in:** `export const maxDuration = 1800` per function. Project-level defaults above 800s are not yet supported in the beta — must be per-function.
- **Billing:** Fluid Compute bills active CPU only; I/O waits (LLM calls, DB queries, third-party APIs) are free.
- **Edge runtime:** not covered. The 25s-response / 300s-stream cap on Edge is unchanged. The control plane (app.deessejs.com) is on Edge and remains inside its existing envelope.

**Impact on C1:** Risk #5 is gone. The 17.13/17.14 streaming agent loop that motivated Risk #5 now fits inside a single Function invocation. The 4-6 min end-to-end target is **not** constrained by function duration anymore. C1's new floor is controlled by the **Vercel + Trigger.dev build wall time**, not the agent-loop duration. Re-evaluated floor estimate: see C1.7.

Source: https://vercel.com/changelog/vercel-functions-can-now-run-up-to-30-minutes and https://vercel.com/docs/functions/configuring-functions/duration (last_updated 2026-05-14).

With **sequential mints**, sequential deploys (60-180s combined), and buyer click (60s avg): **~225-300s ≈ 3.75-5 min.** This matches the feasibility doc's 4-6 min range. **The wins below compress it to 75-180s.**

### C1.3 — Trigger.dev 90-150s build: yes, it's the bottleneck. Pre-caching halves it.

Decomposed:

- **Image pull:** ~5-10s (operator pre-builds base image; layers reused).
- **`pnpm install` (cold):** ~20-40s. **Warm with Turborepo remote cache: ~3-5s.**
- **Buildkit layer cache + Turborepo remote cache:** 5-15s incremental. Vercel caches dependencies by default (https://vercel.com/docs/builds/managing-builds). Trigger.dev's Dockerfile builds reuse the same cache if the operator pins a base image and mounts the Vercel remote cache via `VERCEL_REMOTE_CACHE_AUTH`.
- **Code compile / bundle:** 5-10s for DeesseJS template.
- **Image push + Trigger.dev deployment registration:** 10-20s.
- **Trigger.dev org queue wait:** 0-60s depending on tenant count.

**Cacheable surface across tenants** (the DeesseJS template is identical for every tenant):

1. **Base image** (Node 22, pnpm, Trigger.dev SDK, DeesseJS template) — one push, reused. Saves 10-20s/build.
2. **`pnpm` store cache** — Turborepo remote cache on Vercel + Trigger.dev reuses it. Saves 15-35s/build.
3. **Next.js build output cache** (`.next/cache`) — deterministic per commit. If the operator's CI uploads the prebuilt artifact, every tenant's Vercel deploy can skip the build step entirely. Same model Vercel uses for Deploy Button templates.
4. **Trigger.dev Dockerfile layer cache** via the operator's registry.

**Irreducible floor with full caching: ~30-45s.** Without caching: 90-150s. v0 plans 45-60s warm, 120s cold. **The current 4-6 min budget assumes cold path.** With caching, the Trigger.dev build is no longer the floor.

**Vercel build (separately): 30-60s on Next.js Fluid Compute with cache; 60-120s cold.** With Turborepo warm, drops to 15-25s + queue overhead. **The two builds (Vercel + Trigger.dev) can run in parallel from the operator's CI workers — this is a major win (C1.5).**

### C1.4 — Buyer-side "Deploy to Vercel" step: fully skippable via Vercel env var injection

**Yes, fully.** The Vercel REST API `POST /v11/projects` accepts an `environmentVariables[]` array at project creation:

```
environmentVariables: [{ key: "DEESSEJS_SECRET", value: <jwt>, target: ["production","preview"], type: "sensitive" }]
```

(Schema: `{ key, value, target, type, gitBranch? }` with `type: encrypted | sensitive | plain | system`. Source: https://vercel.com/docs/rest-api/projects/create-a-new-project.)

**v0 plan:**

1. Control plane mints the JWT (Step 6).
2. Control plane calls `POST /v11/projects` **with** `environmentVariables[]` and `gitRepository` set to the operator's `deessejs-cloud-template` repo.
3. Vercel auto-imports the repo, runs the first build with `DEESSEJS_SECRET` already in the environment. The deployed app's `/api/internal/boot` exchange succeeds on the first request.
4. The buyer's "Deploy to Vercel" button is replaced by a "View your app" link to the auto-deployed URL.

**Eliminates:**

- 30-90s+ of buyer click latency (often hours).
- "I bought it, now I have to do DevOps" friction (P0 need #2 in feasibility).
- A class of support tickets ("I forgot to paste the secret," "I pasted it in the wrong environment").

**Costs:**

- Operator's Vercel team owns per-tenant Vercel project (already the v0 plan; no new constraint).
- Operator handles custom domain via `POST /v10/projects/{idOrName}/domains` (parallel to build). v0 accepts `*.tenants.deessejs.com` subdomains via Cloudflare wildcard A/AAAA (already in feasibility).
- **`DEESSEJS_SECRET` rotation** becomes a `PATCH /v9/projects/{id}` with new `environmentVariables[]` (30-60s rebuild per rotation). At 365-day expiry and 30 tenants: ~1 rebuild per 12 days. At 1000 tenants: ~3/day. Negligible.

**This is a free 30-90s win and removes a major buyer-friction surface. Ship in v0.**

### C1.5 — Parallel dependency graph: 14s sequential → 5s parallel

Steps 2-5 (Vercel, Turso, Upstash, Trigger.dev) share no inputs and no outputs. **Run in parallel with `Promise.all`.** The mint time drops from 14s → 5s (the slowest of the four). Step 7's Vercel + Trigger.dev deploys also parallelize across two CI workers.

```
t=0.0s  : Step 1 completes (signup webhook arrives)
t=0.1s  : Promise.all starts Steps 2, 3, 4, 5
t=2.0s  : Step 4 (Upstash namespace) resolves     [1.9s]
t=3.0s  : Step 5 (Trigger.dev project) resolves   [2.9s]
t=4.0s  : Step 3 (Turso DB + token) resolves      [3.9s]
t=5.0s  : Step 2 (Vercel project + env var) resolves  [4.9s]
t=5.1s  : Step 6 (sign DEESSEJS_SECRET) — <0.1s
t=5.1s  : Promise.all starts 7a (Vercel deploy) + 7b (Trigger.dev deploy)
t=~50-65s : max(va, tb) resolves — Vercel ~30-60s warm, Trigger.dev ~45-60s warm
t=~60-70s : Step 9 (Function cold start) — 5-10s
t=~62-72s : Step 10 (runtime smoke) — 1-2s
t=~75s  : Tenant "ready" — total wall time
```

**v0 Standard target: 75-90s warm, 150-180s cold.** Both well under 2 min.

### C1.7 — 2-min target: achievable. v0 plan = 75-90s warm, 150-180s cold.

**Yes, achievable.** Both the warm and cold paths are under 2 min. The original 4-6 min estimate is dominated by three things the v0 design has not yet accounted for:

1. **Sequential mints** (14s → 5s with `Promise.all`).
2. **Buyer click latency** (30s-infinity → 0s with auto-inject via Vercel `environmentVariables[]`).
3. **Sequential Vercel + Trigger.dev deploys** (60-180s → 30-90s with parallel CI workers).

The remaining floor is controlled by:

- **Vercel first-build cold path** (15-30s with cache, 60-90s without). v0 should publish a heavily-cached template (Turborepo remote cache, minimal `pnpm install` footprint).
- **Trigger.dev first-build cold path** (45-60s with cache, 90-150s without). Same fix.
- **Vercel Function cold start on a fresh project** (3-8s; included in the 75-90s warm budget).

**Irreducible minimum:** ~60-75s end-to-end wall time on the warm path with full parallelization.

**v0 plan target:** **2 min end-to-end, p50** with the parallelized flow. p95 will be 3-4 min during the first 30 days as the cache warms.

**What the 2 min does NOT include:**

- Buyer's domain verification (DNS propagation, 0-600s; runs async while buyer uses `*.tenants.deessejs.com`).
- Stripe's webhook delivery latency (already counted in 75-90s).

**What we lose to hit 2 min:** two parallel build workers per onboarding event. At $0.0035/CPU-minute (https://vercel.com/docs/builds/managing-builds), ~$0.01-0.05 per tenant. **Net: ~$0.05 per tenant to save 60-120s of wall time. Worth it.**

### C1.8 — Vercel 12-concurrent-builds ceiling is a fleet constraint, not a per-tenant one

The 12-slot pool is **per-team-account** (https://vercel.com/docs/builds/build-queues). v0's 30-tenant cap means we hit this ceiling only during onboarding bursts. At 90-180s/tenant serial, 12 concurrent = 480-960 tenants/day ceiling — 4-8x the v0 target. **Not the v0 bottleneck.** It IS the v1 bottleneck (per C2). The C1 latency fix (2 min) and the C2 fleet throughput fix (Enterprise plan or wave-staged rollouts) are decoupled. Feasibility Risk #1 mitigation is correct; C1 work does not affect it.

### C1.9 — Recommendations for v0 (C1-specific)

1. **Implement `Promise.all` for the four mint operations** (Vercel, Turso, Upstash, Trigger.dev). Saves ~9s. Use `Promise.allSettled` so one provider failure doesn't block the others — fall back to a retry queue with exponential backoff.
2. **Use Vercel `environmentVariables[]` on `POST /v11/projects` to inject `DEESSEJS_SECRET` at mint time.** Eliminates the buyer click step. First email becomes "View your app" not "Deploy to Vercel."
3. **Run Vercel deploy and Trigger.dev deploy in parallel from the operator's CI.** Two workers per onboarding event, ~$0.05 per tenant. Saves 60-120s wall time.
4. **Pre-warm the build cache for the DeesseJS template.** The first commit to `deessejs-cloud-template` is the cold path; every subsequent commit is warm. v0 Standard onboarding = ~75-90s, not 150-180s.
5. **Measure Trigger.dev build time in production.** 90-150s is a documentation estimate. Real p50/p95 should be measured after the first 10 tenants. If p95 > 180s, move Trigger.dev code into a Vercel Next.js Route Handler using `waitUntil` + Upstash Realtime (same architecture intent as 4.9). Trade-off: less isolation, but eliminates the 45-60s Trigger.dev build entirely. v0 target drops to **~30-45s warm / 60-90s cold**.
6. **Vercel Function 1800s update: include 17.13/17.14 long agent loop in `/api/internal/agent-loop` with `export const maxDuration = 1800` and `fluid: true` in `vercel.json`.** Long agent loops no longer need to enqueue as Trigger.dev jobs — they can run inline with Fluid Compute's active-CPU-only billing. **Re-evaluate B2's 4.x job architecture with this on the table.**
7. **Synthetic warm-up:** control plane sends a `GET /` to the deployed URL as soon as the Vercel deploy completes (before sending buyer email). Warms the function instance; buyer's first click is a warm 200-500ms response. Cost: 1 Function invocation per tenant. **Recommended for v0 — perceived latency win is huge.**

## Section C2 — Vercel 12-Builds Ceiling Math

**Author:** C2 specialist. **Scope:** validate the feasibility doc's claim that 12 concurrent builds on Pro caps onboarding at ~480 tenants/day, chart the trajectory at 30/100/1,000 tenants, and document mitigations.

### 1. Concurrent builds: what 12 actually means

The 12-concurrent-builds limit is a **per-team-account slot pool**, not per-project or per-region. Vercel's Build Queues doc is explicit:

> "Hobby accounts allow one build at a time. Pro accounts support up to 12 simultaneous builds. Enterprise accounts can have custom limits based on their plan."

(Source: https://vercel.com/docs/builds/build-queues; cross-referenced in https://vercel.com/docs/limits under "Concurrent Builds".)

**Implication for DeesseJS Cloud:** every tenant on the operator's Vercel Pro team account shares the same 12-slot pool. A template update that fans out to 100 tenants in one push would queue 88 builds immediately. This is the binding constraint being modeled.

**On-Demand Concurrent Builds (the escape valve):** Vercel sells project-level on-demand concurrency that "dynamically scales the amount of builds that can run simultaneously" so "builds will never queue." Two modes: `SKIP_NAMESPACE_QUEUE` (all builds run immediately) or `WAIT_FOR_NAMESPACE_QUEUE` (one active build per branch). On-demand concurrency is **metered separately** — build usage is priced at $0.0035 per CPU minute, and Elastic build machines are billed by CPU minute ($0.0035/CPU-minute starting). Standard machines are only billed when on-demand concurrency is enabled. (Source: https://vercel.com/docs/builds/managing-builds, https://vercel.com/docs/pricing#builds.)

### 2. Build duration distribution

The feasibility doc estimates 90–150s per build. Without a direct benchmark on the DeesseJS v1 template, the realistic distribution for a Next.js + pnpm + 23-surface SaaS template on Vercel is:

| Percentile | Estimate | Notes |
|---|---|---|
| p50 | **75–95s** | pnpm install + Next.js build with build cache hit (the second build onward on the same template hash) |
| p95 | **140–180s** | Cold pnpm install (no cache), all 23 surfaces compile |
| p99 | **220–300s** | Cold install + a slow route + a flaky network fetch in a build step |

A 150s estimate is **slightly pessimistic but realistic as a worst-case planning figure**. Caching is the dominant lever: the operator's CI pipeline should pre-warm the Vercel Build Cache (`vercel.json` cache directive, `pnpm` store cache) before any tenant onboarding run, so the buyer-facing build is always cache-hit. **Realistic p50 with warm cache: 60–80s.** Realistic p50 cold: 110–130s.

**Build ceiling is 45 minutes per deployment** (https://vercel.com/docs/limits#build-time-per-deployment). The 90–150s range is well under this, so build duration itself is not a constraint.

### 3. Throughput math: where is the 14x gap?

The feasibility doc says ~480 tenants/day. Raw saturation says 12 × (86,400 / 150) = **6,912 tenants/day**. The 14.4x gap comes from:

1. **Builds don't happen continuously** — they happen in bursts triggered by signup. A signup spike of 200 signups in an hour would queue 188 immediately.
2. **The 12-slot pool is shared with platform updates** — every time the operator pushes a template update, that fan-out also consumes slots. A 100-tenant fan-out blocks all 12 slots for ~12 minutes.
3. **The funnel has friction** — signup → payment → DNS → buyer's "Deploy to Vercel" click is not instantaneous. Real throughput is gated by the slowest step in the funnel, not the build.
4. **Other rate limits** — the deploy hook triggers rate limit is **60 per hour per owner** (https://vercel.com/docs/limits#rate-limits). The Deployments per hour limit on Pro is 450 (https://vercel.com/docs/limits#rate-limits), which at 150s/build matches 12 concurrent × 3600/150 = 288/hour, leaving 162/hour of headroom. The deployments per day limit on Pro is **6,000**.

**Corrected throughput model for 480 tenants/day:**
- 12 concurrent builds × (86,400 / 150) = 6,912/day theoretical max
- × 0.50 effective utilization (signups spread across 24 hours, not a constant queue of 12) = **3,456/day**
- × 0.14 acceptance rate (only ~14% of signups actually click the "Deploy to Vercel" email button within the first 24 hours) = **480/day**
- × 0.5 buffer for operator template updates consuming the same slots = **240–480/day practical ceiling**

The 480/day figure is reasonable for the v0 30-tenant beta but **optimistic for a 1,000-tenant/mo public launch**.

### 4. Trajectory: when does 12 become binding?

| **30 tenants (v0 beta, 2026-Q3)** | ~5–10 builds/day (signups + occasional updates) | ~1% | **No** — trivial headroom |
| **100 tenants (early traction, late Q3)** | ~20–40 builds/day + 1 weekly template fan-out (~100 builds) | ~10% daily; ~80% during weekly fan-out | **No** — fan-out takes 50–60 minutes at 150s each; users see a delay but no failure |

### 5. Mitigations and trigger points

| Mitigation | What it does | Trigger to enable | Cost |
|---|---|---|---|
| **Wave-staged rollouts** | Push template updates in batches of 12 tenants at a time, with health gates between waves | v0 (default) | Free — operational |
| **Deferred deploy queue** | New tenant gets a "queued for deploy" status; operator dashboard deploys in waves during off-peak hours | v0 if beta signups cluster | Free — operational |
| **Deploy-on-first-request** | Skip the email-deploy step entirely; tenant URL serves a 30s "preparing your app" splash until first request triggers the build | v0 if buyer friction is measured | Free — adds Tailwind splash page + edge function |
| **On-Demand Concurrent Builds** (Vercel project-level) | Skip the queue for specific projects; pays $0.0035/CPU-minute | When queue waits exceed 30 min | Variable — ~$0.05 per 150s Standard build × concurrent slots used |

### 6. Vercel Enterprise ceiling — what does it actually raise?

### 7. Critical cross-check: 12-build ceiling for platform-update fan-out

The feasibility doc's flow step 11 ("push to deessejs-cloud-template triggers Vercel deploy hook that fans out to all tenants") is **the bigger risk than onboarding**. ## Section A4 — Upstash Shared-DB Noisy-Neighbor Reality

**Author:** A4 specialist. **Scope:** validate feasibility Risk #3 (shared Upstash noisy neighbor) with hard evidence, document the per-key vs per-command Ratelimit distinction, model the shared Global Database blast radius, evaluate Turso as a per-tenant counter store, price the shared-vs-dedicated inflection, and design the 60-second poison-pill kill switch.

### A4.1 — Risk #3 is real, and worse than the feasibility doc states

The feasibility doc says "a single misbehaving tenant degrades Upstash for all." After documenting Upstash's actual primitive behavior, this risk is **confirmed and amplified**:

1. **Ratelimit prefix is a Redis key namespace, not a tenant budget.** `prefix: "t:tenant-42:"` only avoids collisions when the operator shares Redis across applications; it does **not** aggregate commands across all keys for one tenant into a shared bucket. Each call to `ratelimit.limit(identifier)` tracks that identifier independently (Upstash Ratelimit docs, prefix description: "Optional prefix for the keys used in redis. This is useful if you want to share a redis instance with other applications and want to avoid key collisions. The default prefix is '@upstash/ratelimit'"). **The feasibility doc's claim that "the Ratelimit `prefix` throttles per-key, not per-command" is correct — but understated. The SDK is identifier-scoped; there is no built-in tenant-wide aggregate limit mode.**
2. **The shared Global Database has no per-tenant isolation at all.** All tenants share the same Redis instance via key prefixes. The blast radius of one tenant burning commands is "everyone on the shared DB feels it." Upstash markets Global Database as "infinitely scalable" and "99.99% uptime guarantee with auto-scaling" — but does not promise per-key or per-prefix throughput isolation. The auto-scaling is database-wide.
3. **Per-database commands/sec cap exists, and it is small.** Free, Pay-as-you-go, and Fixed tiers cap at **10,000 commands/sec** per database (100 GB and 500 GB Fixed tiers cap at 16,000). Past that, Upstash rate-limits the database instead of continuing to bill. Source: https://upstash.com/docs/redis/overall/billing.
4. **The cost of "rate-limited instead of billing" is a full outage for all tenants on the database.** When the per-database cap is hit, every tenant's commands fail until traffic drops. There is no per-tenant pre-warning or graceful degradation.

**Risk reclassification:** feasibility Risk #3 is not a single-tenant cost-control concern; it is a **fleet-wide availability risk** with a known hard ceiling (10k CPS per shared database) and no native per-tenant rate-limit primitive.

### A4.2 — The 60-second kill switch: which mechanism is actually fastest?

Four candidates, ranked by propagation latency:

| Mechanism | Endpoint / API | Time-to-effect | Notes |
|---|---|---|---|
| **A. Rotate JWT in Turso (`control_plane.tenant_status = 'suspended'`)** | Operator updates Turso row → Vercel Function `/api/internal/boot` checks status on next invocation | **5-15s** (next request, no Vercel propagation) | Each request to `/api/internal/boot` re-validates the JWT. Setting `tenant_status='suspended'` returns 403 on the next call. Cold-start functions hit the same path. **Recommended** — applies to all entry points (web, API, jobs). |
| **B. Rotate Upstash REST token** | `PATCH /v1/redis/{id}/rotate-token` (if exists) or delete + recreate | **30-60s** | Requires Upstash API for token rotation. Docs do not document a rotate endpoint; manual reset via console. **Slow.** |
| **C. Vercel Trusted IPs** (block tenant IP) | `PATCH /v9/projects/{idOrName}` with `trustedIps` set to exclude tenant | **30-90s** (CDN cache TTL) | Per-tenant IPs are not stable; Cloudflare egress IPs vary. Block list grows unboundedly. **Insufficient — wrong abstraction.** |
| **D. Vercel WAF IP Blocking** | `POST /v1/security/firewall/bulk` with custom rule | **30-90s** (CDN edge propagation) | Same problem as C: tenant IPs are not stable. Block list grows. |
| **E. Vercel `passwordProtection` on the project** | `PATCH /v9/projects/{id}` with `passwordProtection` set | **5-30s** (CDN cache) | One-shot kill; requires Vercel rebuild to undo. **Effective** but coarse — affects all routes including the buyer's own admin UI. |
| **F. Vercel `DELETE /v9/projects/{idOrName}`** | Immediate | **5-30s** (CDN propagation) | **Destroys state** — recover is operator-side restore from Turso + Upstash backup. **Last resort.** |
| **G. `DEESSEJS_SECRET` revocation list** in Vercel KV (the JWT carries a `kid` and the operator revokes it) | Operator writes revocation flag; boot handler checks `kid` against KV | **5-15s** | Same architecture as A but uses Vercel KV instead of Turso. **Equivalent.** |

**Recommended design:** mechanism **A** — a `tenant_status` column in the operator's Turso DB, updated synchronously when the kill switch fires. The `/api/internal/boot` handler reads it on every invocation; on `suspended` or `killed`, it returns 403. Combined with **E** (Vercel `passwordProtection`) as a belt-and-suspenders backup for cases where the operator wants to stop traffic at the edge without waiting for the Vercel Function to warm up.

**End-to-end kill latency: 10-30s** for the Vercel Function path, **5-30s** for the Vercel CDN path. The 60-second target in the feasibility doc is **achievable** with this design.

### A4.3 — Turso as the per-tenant counter store: latency math

The feasibility doc proposes "per-tenant daily command counter enforced in Turso (not Upstash)" to police noisy-neighbor behavior. Question: is Turso fast enough to be on the hot request path?

**Turso HTTP latency (best available evidence):** the official docs do not publish p50/p99 numbers for primary SELECTs over HTTP. The closest evidence is:

- **Embedded replica reads:** "microseconds" from local file (https://docs.turso.tech/features/embedded-replicas) — applies when the DeesseJS runtime is colocated with the replica. Vercel Functions run in `iad1` by default (https://vercel.com/docs/functions/limitations), and a Turso replica in `us-east-1` (Virginia) is a 1-2ms HTTPS round-trip from there.
- **Write semantics:** writes go to the primary first, then propagate to replicas. "After a write returns successfully, the replica that initiated the write will always be able to see the new data right away, even if it never calls `sync()`" — but cross-replica read-after-write is eventual.
- **Typical Vercel Function cold start:** 3-8s (per C1.7). On a warm Function, the request path is dominated by application logic; a Turso round-trip is **5-30ms** in practice based on community benchmarks (no published Upstash/Turso benchmark found in this research; numbers are engineering estimates).

**Counter pattern viability:**

```ts
// On every Upstash command:
// 1. INCR counter in Upstash (fast, but doesn't block Upstash access)
// 2. ASYNC: write daily total to Turso (eventually consistent)
// 3. Operator cron: SELECT SUM(commands) FROM t_usage WHERE tenant_id = ? AND day = ?
// 4. If SUM > limit → set tenant_status = 'throttled' → boot handler returns 429
```

**The counter MUST NOT be on the synchronous hot path.** A synchronous Turso SELECT before every Upstash command would add 5-30ms per request and turn every Upstash blip into a Turso outage. The architecture must be:

1. **Upstash INCR** is on the hot path (~1ms, sub-ms p99 in the same region per Upstash's <1ms same-region read latency claim for the Global Database).
2. **Turso write** is a background flush — every 60s, or batched at the end of each request handler in a `waitUntil` block. Cost: 1 Turso write per request, or 1 per 60s aggregation window.
3. **Turso read** for throttle decision is on the cold path (boot handler, JWT refresh, periodic cron).

**This is feasible at v0.** Turso can host the per-tenant counter, but only if it is **eventually consistent** with the Upstash command stream. The 60-second poison-pill latency is consistent with this design (poll Turso every 60s, set throttle flag).

### A4.4 — Pricing tier inflection: shared Global vs per-tenant Regional

The feasibility doc says "shared Global Database" without committing to a specific Upstash tier. The cost math:

**Shared Global Database on Pay-as-You-Go:**
- Per command: **$0.20 per 100K**. With one primary + one read region (Global), writes are replicated → 100K writes = $0.40 (2× cost per write).
- 10K commands/sec cap per database.
- Max 100 databases per account (operator-wide).

**Per-tenant Regional Database on Pay-as-You-Go:**
- Same per-command price ($0.20/100K), no replication overhead.
- Same 10K commands/sec cap.
- **100 databases cap** is the binding constraint: 100 tenants per Upstash account.

**Per-tenant Regional Database on Fixed tier (e.g., 250 MB):**
- **$10/mo flat**, 10K commands/sec cap. No per-command charge.
- Predictable cost, no surprise overage from runaway tenants.

**Per-tenant Regional Database with Prod Pack add-on:**
- **$10 + $200 = $210/mo**. Adds 99.99% SLA, multi-zone HA, encryption-at-rest, SOC-2, Prometheus, Datadog.

**Inflection math at v0 (30-tenant beta):**

| Usage profile | Shared Global DB (1 DB) | 30 per-tenant Regional Fixed 250MB | 30 per-tenant Regional + Prod Pack |
|---|---|---|---|
| Light (1M commands/tenant/mo = 30M total) | $60/mo ($0.20 × 300 = $60) | $300/mo ($10 × 30) | $6,300/mo ($210 × 30) |
| Medium (10M/tenant/mo = 300M total) | $600/mo | $300/mo | $6,300/mo |
| Heavy (100M/tenant/mo = 3B total) | $6,000/mo | $300/mo | $6,300/mo |

**The crossover is at ~5M commands/tenant/mo.** Below that, shared Global is cheaper (and fits in the 10K CPS cap if commands are reasonably distributed across 30 tenants). Above that, per-tenant Fixed is dramatically cheaper and isolates risk.

**At v0 (30 tenants, mostly light usage):** shared Global Database on Pay-as-You-Go is **~$60-600/mo** — negligible vs the rest of the stack. The risk is concentrated: one runaway tenant pushes the shared DB to its 10K CPS cap and **everyone degrades**.

**At v1 (300 tenants, mixed):** the math flips. A single Shared Global DB has a 10K CPS cap → 300 tenants × 33 CPS avg = 10K CPS. **Already at the ceiling with no headroom for spikes.** Per-tenant Fixed at $10/mo = $3,000/mo (no per-command charge, predictable). The shared architecture is **architecturally incompatible** with 300+ tenants.

**Recommendation:** treat the shared Upstash Global Database as a **v0-only architecture**, with a documented migration plan to per-tenant Regional Fixed databases at the **100-tenant trigger** (which is well before the 10K CPS ceiling becomes binding). Per-tenant Prod Pack ($200/mo each) is reserved for Enterprise tenants only.

### A4.5 — The 10K commands/sec cap and what happens when it's hit

Upstash's billing page is explicit: when a Pay-as-You-Go database exceeds the 10K CPS cap (or hits the monthly budget set on the database), it is **"rate-limited rather than continuing to bill."** This is a hard cutover, not a backpressure ramp.

**Failure modes when one tenant burns 100k commands/sec on a shared Global DB:**

1. **Database-wide rate limit kicks in.** All commands from all tenants begin failing. Failure is uniform — every tenant sees the same degradation.
2. **No partial outage.** There is no concept of "tenant A is rate-limited while tenant B continues." The whole DB is throttled.
3. **Recovery is automatic** when the offending traffic drops below the cap. No operator intervention.
4. **No native alerting on per-tenant command volume.** Upstash surfaces database-wide metrics; per-prefix accounting must be built by the operator (via Turso counter pattern in A4.3).

**Implication:** the shared Global Database is safe **only** if the operator enforces per-tenant CPS via the Turso counter pattern and the JWT boot check. Upstash's native primitive gives no protection.

### A4.6 — Per-tenant Upstash CPS enforcement: the implementation pattern

Recommended architecture for v0:

1. **Every DeesseJS service wrapper** that calls Upstash uses a `withMeteredUpstash(tenantId, command)` helper:
   ```ts
   async function withMeteredUpstash<T>(
     tenantId: string,
     fn: () => Promise<T>,
   ): Promise<T> {
     // 1. Check throttle flag (cached in-process, 60s TTL)
     const isThrottled = await bootCache.getThrottleStatus(tenantId);
     if (isThrottled === 'throttled') {
       throw new UpstashTenantThrottledError(tenantId);
     }
     // 2. Execute command
     const result = await fn();
     // 3. INCR counter (Upstash, fast, ~1ms)
     await upstash.incr(`meter:${tenantId}:${today()}`);
     // 4. ASYNC: schedule flush to Turso (don't await)
     waitUntil(flushMeterToTurso(tenantId));
     return result;
   }
   ```

2. **Boot handler** (`/api/internal/boot`) checks Turso `tenant_status` and reads `t_meter` to compute rolling CPS. If rolling CPS exceeds the per-tenant limit (configurable, default 100 CPS for Standard / 500 CPS for Studio), set `tenant_status = 'throttled'`.

3. **Upstash counter is the source of truth for command volume** (eventually consistent with Turso). Turso counter is the **billing + throttle decision** store.

4. **Latency budget:** add `~1-2ms` per command for the INCR. Acceptable. Flush to Turso is fire-and-forget.

### A4.7 — Recommendations for v0 (A4-specific)

1. **Adopt the shared Global Database for v0** (≤30 tenants, ≤10K CPS fleet-wide, ≤$60/mo) — confirmed cost-effective.

## Section B1 — Trigger.dev: necessary or over-spec?

**Author:** B1 specialist. **Scope:** after the 2026-06-15 Vercel Function 1800s update, re-evaluate whether Trigger.dev is the right queue primitive for v0 (30 tenants) and v1 (1000 tenants), or whether the locked stack's other primitives (Vercel Queues, Vercel Workflows, Vercel Cron, Vercel Functions + Fluid Compute, Upstash Realtime) cover the workload without adding a vendor. Compares Trigger.dev v4 Cloud, Inngest, Temporal, and Vercel-native primitives on DX, retries, observability, and cost. Documents what the 4.9 "long-running tasks" wiring should actually look like.

### B1.1 — TL;DR

1. **The 30-min Vercel Function ceiling does not eliminate the need for a queue primitive.** It reduces the *floor* of what must be offloaded, but every other reason Trigger.dev was justified still stands: retries with exponential backoff, dead-letter queue, observability dashboard, per-tenant rate limiting, webhooks (in and out), cron, and the `send.at(...)` delayed-job API. All of those still need a runtime — a Vercel Function alone does not provide them.
2. **A new option now exists in the locked stack that the feasibility did not consider: Vercel Queues + Vercel Workflows.** Shipped GA 2026-04-16 (Queues) and 2026-05-20 (Workflows). They run inside the Vercel project, so they do not require a separate "Trigger.dev per-tenant project" in the control plane. This is a real change.
3. **For v0 (30 tenants): Trigger.dev is over-spec.** Vercel Queues + Vercel Workflows, scoped per-tenant by Vercel project ID, deliver 80%+ of Trigger.dev's capability for the 4.x job categories, at zero added per-tenant control-plane surface, with one fewer vendor relationship. The lost 20% (long-horizon steps that exceed 30 min and the deep AI-agent tracing) can be approximated with a chained Workflow + Upstash Realtime pattern, which is what 4.9 should look like anyway.
4. **For v1 (1000 tenants): the answer is the same.** Trigger.dev's per-tenant project model is the wrong shape for a 1000-tenant SaaS; the operator would need to provision and maintain 1000 Trigger.dev projects, on its own org's billing, with concurrency and compute limits shared across all of them. Vercel Workflows scales with the Vercel project; that maps 1:1 to a tenant and removes the cross-tenant resource contention risk by construction.
5. **The job categories that still genuinely need a separate queue primitive** (AI agent runs that exceed 30 min, long-horizon batch ingestion for 17.20 RAG when it lands, scheduled exports, data-portability weekly jobs) are below 5% of v0 traffic. For those, **self-host Trigger.dev v4 on a single operator-owned project** (not per-tenant) and invoke it from each tenant's Vercel Workflow. This is a 5-tenant-month-of-engineering add-on, not a v0 requirement.

This section replaces feasibility Risk #5 entirely. The 300s ceiling is gone; the new ceiling is the Vercel Function's 1800s extended-max-duration beta with Fluid Compute, and a separate "long job" escape hatch (Trigger.dev self-host) is reserved for the > 30 min case only.

### B1.2 — Re-categorization of 4.x workloads after the 30-min ceiling

The feasibility doc's Risk #5 said AI features "must enqueue long agent runs as Trigger.dev jobs (4.9 already wired) and stream back via Upstash Realtime." That sentence survives the Vercel update, but the *threshold* moves from 300s to 1800s, and the per-workload routing is now:

The Trigger.dev product was justified because (a) it bundled all 13 in one runtime, and (b) the 300s ceiling forced 4.9 onto a queue. With the ceiling gone, reason (b) weakens; reason (a) is now matched by the Vercel stack.

**17.x AI workloads specifically.** The 17.13 multi-step agent loops and 17.14 human-in-the-loop checkpoints are the long-horizon workloads that *originally* justified 4.9 being a job. Empirically (the feasibility Risk #5 cites this), they can hit 300s. They can also stay under 1800s for nearly all real use cases if `maxSteps` is bounded and tool calls are short. The case where they exceed 1800s is the "agent that processes 1000 documents in a loop" — a v2 (17.20 RAG) workload, not v0.

**5.x mail workloads.** 5.13 ("Email queue, batching, throttling") reads as "Resend native + our queue on Trigger.dev" in the v1 feature doc. With the Vercel stack, the same is "Resend native + our queue as a Vercel Queues topic named `mail`." Resend has its own queue; the 5.13 layer is a tenant-side throttling concern, which Vercel Queues solves with `maxConcurrency` per consumer group.

**11.x notifications.** 11.1 ("Real-time delivery via Upstash Realtime") is already the chosen path. The job that *publishes* to Upstash Realtime is what we are routing in this section. That job is < 1s, fits in a Vercel Function, and the only reason to put it on a queue is to absorb traffic spikes — exactly Vercel Queues' use case.

### B1.4 — Trigger.dev v4 per-tenant projects — does it actually work?

The feasibility picks Trigger.dev v3 partly because it supports "per-tenant projects on a shared org." The intent is operator isolation: each tenant gets its own project, its own env, its own concurrency, its own billing boundary.

**Confirmed facts:**

- Trigger.dev v4 cloud supports **multiple projects per org** (the `trigger.dev` CLI has `trigger dev` and `trigger deploy --project=<ref>`). The control plane surfaces projects in the dashboard.
- Each project has its own **environment variables** and **API keys** scoped to the project.
- The **Hobby plan ($10/mo) includes 50 concurrent runs across the whole org**; the **Pro plan ($50/mo) includes 200 concurrent runs across the whole org**. This is the key constraint.
- Compute is **per-second** (Small 1x at $0.0000338/sec is the default), billed at the org level. Per-tenant cost isolation is therefore a *billable usage* line, not a hard budget. The operator can attribute cost per project by tagging each job run with `tenantId`, but cannot hard-cap a single project without engineering work (the `concurrency` per-queue config caps concurrency, not cost).
- Schedules, alerts, and team members are **per-org caps** (Hobby 100 schedules, Pro 1,000+, etc.).
- The management API exists but the operator would need to script project creation, env var set, and project deploy against the API. This is a real engineering tax.

**Per-tenant env isolation reality check:** the per-project env var is set at deploy time. If the operator wants `STRIPE_SECRET_KEY` to differ per tenant (e.g., for Stripe Connect Standard, the buyer's connected account credentials), it must deploy a separate version of the worker code per tenant. The feasibility doc does not address this. The simplest workaround is to pass `tenantId` and the per-tenant secrets *in the job payload*, fetched from a sealed Upstash Redis key at job start, decrypted in memory. That works but adds latency to every job.

**At 30 tenants:** a single Trigger.dev Cloud Pro org at $50/mo + 30 per-tenant projects (env isolation) + 200-concurrent cap shared across all 30 = **$50 + usage** (~$0.10/tenant/mo compute at low band) = **~$60/mo total**. Feasible.

**At 1000 tenants:** a single Pro org's 200-concurrent cap is **0.2 concurrent runs per tenant** — a single AI-heavy tenant doing 20 parallel agent loops would consume 10% of the entire org's concurrency budget. The 200-cap gets shared across 1000 tenants, with no per-tenant fairness guarantee. The operator would need to:
1. Move to Enterprise pricing (custom, almost certainly $1k+/mo).
2. Manually implement per-tenant rate limiting at the enqueue site (in the Vercel Function that triggers the Trigger.dev job).
3. Accept that one misbehaving tenant can starve the others.

This is a real, well-understood multi-tenant queue problem. It is the **same shape** as the Upstash shared-DB noisy-neighbor risk in feasibility Risk #3 (and detailed in Section A4 of this synthesis). The mitigation is per-tenant primitives by construction, not shared by billing. Vercel Workflows gives you that for free (one Vercel project per tenant = one queues/workflows namespace per tenant = no shared rate-limit surface). Trigger.dev v4 Cloud does not.

**Self-host Trigger.dev v4 on a single project:** the alternative. One Trigger.dev project on the operator's own infrastructure, dispatching jobs by `tenantId` in the payload. No per-tenant projects, no per-tenant env, no shared concurrency cap (you set the cap). Cost is the infra bill: $50-200/mo for the box(es), $0 from Trigger.dev. This is the **right answer for a 1000-tenant DeesseJS Cloud** if you decide Trigger.dev is the right product. The catch is operational: you own the SLA, the upgrade cadence, the Redis/Postgres/worker scaling, and the on-call rotation. The Trigger.dev self-host docs are explicit: *"You are fully responsible for security, uptime, and data integrity."*

### B1.5 — The 4.9 "long-running tasks" wiring — what it actually looks like

The feature doc 4.9 is checked with the note *"Trigger.dev v3 supports."* The feasibility doc refers to *"4.9 already wired"* without describing the implementation. The locked stack also has Upstash Realtime (11.1) and Vercel AI SDK (17.1). The actual wiring for an AI agent run that streams back to the user should be:

**Path A — Vercel-native (recommended for 90% of v0 jobs):**

1. The user's browser opens a Server-Sent Events stream to a Vercel Function route `app/api/ai/run/route.ts` with `export const maxDuration = 1800` and Fluid Compute enabled. The function streams via `ReadableStream`.
2. The function runs `generateText` (Vercel AI SDK) with `maxSteps` bounded to a safe number (e.g., 25). For most use cases this completes in < 60s and the SSE response closes.
3. For long jobs that exceed the function's safe `maxDuration`, the function kicks off a **Vercel Workflow** via `start()` from `workflow`, returns a `runId` and a stream-subscription URL to the client immediately, and closes the SSE. The client subscribes to a separate streaming endpoint that reads Workflow's `getRun()` stream.
4. Per-tenant cost tracking (17.15), per-tenant rate limit (17.16/17.17), and per-tenant token budgets are enforced *inside the Vercel Function* before the AI SDK call, in Turso (per-tenant config rows) and Upstash (per-tenant counter). This is the existing DeesseJS 17.x architecture and survives the Trigger.dev re-evaluation unchanged.

**Path B — Upstash Realtime hybrid (for jobs that need cross-function fan-out):**

1. A Vercel Function receives the request, validates per-tenant budget (17.16), and enqueues a Vercel Workflow. Returns `202 Accepted` with a stream URL.
2. The Workflow's steps publish progress to Upstash Realtime channels keyed by `t:{tenantId}:ai-run:{runId}` and `t:{tenantId}:user:{userId}`.
3. The browser subscribes to the per-user channel via the `useRealtime` hook (11.1). The Upstash Realtime backing store is Redis Streams with `history.maxLength` bounded to a sensible number (e.g., 1000) and `history.expireAfterSecs` of, e.g., 3600.
4. The Workflow is the *source of truth*; Upstash Realtime is the *delivery layer*. A dropped SSE connection does not lose data — the client re-subscribes and replays from history.

**Path C — Trigger.dev v4 self-host fallback (for jobs > 30 min):**

1. Same as Path A or B through step 2.
2. The Workflow's "long step" is a `step.run()` that calls the operator's self-hosted Trigger.dev REST API (`POST /api/v1/tasks/trigger`) with `{ task: "ai-agent-long", payload: { tenantId, runId, ... } }`.
3. The Trigger.dev worker runs the agent, periodically POSTs progress to Upstash Realtime via the same channel naming.
4. The Trigger.dev run's `runId` is the same as the Vercel Workflow's external ID; both surfaces expose status. The Trigger.dev dashboard is the deeper observability view for the operator.

**This is what "4.9 already wired" should mean.** The feasibility's framing — "Trigger.dev + Upstash Realtime" — survives intact, but the Trigger.dev role is reduced to a fallback for the long-tail, and the per-tenant boundary is the Vercel project, not a Trigger.dev project.

### B2.3 — Streaming primitive for 17.13/17.14: the comparison

The DeesseJS v1 spec for 17.10 ("Token streaming") and 17.19 ("Streaming UI primitives, drop-in components") says the floor is **Vercel AI SDK native streaming**. For 17.13 (multi-step agent loops) and 17.14 (human-in-the-loop checkpoints), the buyer-facing surface is a streaming UI element (chat bubble, progress indicator, approval modal). The technical question is **what is the wire format and transport** that carries tokens, tool-call boundaries, and pause/resume signals from the Vercel Function (or Workflow) to the browser.

Four candidates, ranked against the locked stack:

| Primitive | Wire format | Transport | Pros | Cons |
|---|---|---|---|---|
| **Vercel AI SDK `streamText` + `useChat`** | AI SDK Data Stream Protocol (newline-delimited JSON, chunked) | HTTP chunked transfer from the Function, consumed by `useChat` on the client | Native to Vercel AI SDK; drop-in component in 17.19; no extra infra; works in the existing Function route without a queue | Drops on connection loss unless re-subscribed; only works for the open connection lifetime; no replay history; no multi-channel fan-out |
| **Server-Sent Events (SSE)** | Custom event format (typed events: `token`, `tool_call`, `done`, `error`) | HTTP `text/event-stream` from the Function, EventSource API on client | Simple; works behind Vercel's CDN; native `ReadableStream`; 30-min ceiling now allows long agent runs without reconnect | No binary; one-way; single connection; no replay history without external store |
| **Upstash Realtime (Redis Streams + SSE)** | JSON messages validated by Zod schema, broadcast over SSE | `realtime.channel().emit()` on server, `useRealtime(channel)` on client | Survives connection drops (`history.maxLength` + `history.expireAfterSecs`); multi-channel fan-out (one channel per run, one per user, one per org); Zod-validated payloads; works for the HITL checkpoint pattern (one channel for status, one for events) | Second vendor dependency on the hot path; cost is per-message; latency adds ~5-15ms vs raw SSE |
| **Vercel Workflows `getRun()` stream** | Workflow Run Stream (chunked JSON, Workflows-specific schema) | Long-poll or SSE from `workflow.getRun(runId)`; subscribe via Workflows SDK | Survives Function instance recycling; durable across deploys; replay from any step; native to Workflows' `waitForEvent` pattern (17.14) | Locked to Workflows (no portable stream surface if we migrate off Vercel); no built-in multi-channel fan-out (you'd still need Upstash Realtime for the "user sees the same run on two devices" case); 7-day retention on Pro (matches Trigger.dev) |

**Pick: Upstash Realtime as the delivery layer + Vercel AI SDK's `streamText` as the producer.** Reasoning:

1. **The hot path is already on Upstash Realtime for 11.1 in-app notifications.** Reusing the same channel infrastructure for 17.13/17.14 progress means one vendor to monitor, one auth pattern, one client SDK (`useRealtime`) to teach the buyer.
2. **The "multi-step agent run" requires multi-channel fan-out by construction.** A single run must update the chat UI *and* the in-app notification list *and* the org-level "AI activity" widget. Upstash Realtime channels (`t:{tenantId}:user:{userId}`, `t:{tenantId}:ai-run:{runId}`, `t:{tenantId}:org:{orgId}:activity`) solve this with one emit.
3. **The "survives connection drops" property matters for AI agent runs.** A 5-minute agent run on a flaky mobile network cannot afford to lose progress because the user switched apps. Upstash Realtime's `history.maxLength=1000` + `history.expireAfterSecs=3600` is the durability floor.
4. **Vercel AI SDK's `streamText` is still the producer.** The Vercel Function calls `streamText(...)`, receives an `AsyncIterable`, and pipes each chunk through `realtime.channel().emit()`. The client uses `useRealtime(channel)` for live updates and `useChat` for the *initial* message layout (text bubble, avatar). Two SDKs, one UX.

**For 17.14 HITL specifically:** Vercel Workflows' `waitForEvent` is the orchestration primitive (it pauses the run for up to 1 year waiting for an external event). The "approval modal" UI listens on `t:{tenantId}:ai-run:{runId}:status` via `useRealtime`. When the buyer clicks Approve, the client POSTs to `/api/ai/run/{runId}/approve`, which calls `workflow.sendEvent(runId, "approval", { approverId, decision: "approved" })`. The Workflow resumes from where it paused. **This pattern is impossible to replicate on raw SSE or on Vercel AI SDK streaming alone.**

### B2.4 — The 4.9 surface disposition: stay Trigger.dev, migrate to Vercel Function streaming, or split?

The v1 spec says 4.9 ("Long-running tasks, hours") is Trigger.dev v3 supported. The feasibility doc Risk #5 says "AI features must enqueue long agent runs as Trigger.dev jobs (4.9 already wired) and stream back via Upstash Realtime." Three options:

| Option | What changes | Spec impact | Operational impact |
|---|---|---|---|
| **A. Keep 4.9 as Trigger.dev (per feasibility)** | Nothing — but conflict with B1's recommendation to drop Trigger.dev from v0 | None | Per-tenant Trigger.dev projects, 200-concurrent cap shared across org, 3-min provisioning cost, third vendor |
| **B. Migrate 4.9 to Vercel Function streaming** | 4.9 surface becomes `maxDuration=1800` Fluid Compute Function + Upstash Realtime | Rewrite of 4.9 spec, rewrite of 17.13/17.14 streaming path, deprecation of Trigger.dev code path | Removes Trigger.dev from v0 stack entirely; > 30 min jobs (rare) get rejected with `HTTP 413` |
| **C. Split: ≤ 30 min on Vercel Workflows, > 30 min on self-hosted Trigger.dev v4** | 4.9 spec becomes a routing layer that decides at job-start which runtime to use | Rewrite of 4.9 spec as a router, new spec for the self-host Trigger.dev deployment | Removes Trigger.dev Cloud from v0 stack; adds one self-host Trigger.dev box ($100/mo Fly.io) for the <5% tail |

**Recommendation: Option C (split).** Justification:

1. **The 30 min Vercel Function ceiling is BETA above 800s.** Until the beta lifts, AI-heavy design partners who actually exercise the > 30 min path need a fallback. Option B (no fallback) makes the v0 SLA weaker than it needs to be.
2. **Option C is a strict superset of Option B.** The Vercel-native path is the default; the self-hosted Trigger.dev is the escape hatch. The split is a runtime choice, not an API change for the buyer — same `runId`, same Upstash Realtime channel, same `useRealtime` UI.
3. **Self-hosted Trigger.dev v4 on a single operator project** (not per-tenant) avoids the B1.4 per-tenant project scaling problem. Per-tenant dispatch is via `payload.tenantId` in the task body, with per-tenant secrets fetched from Upstash at job start. This is the same pattern the feasibility doc's "DEESSEJS_SECRET" already establishes.
4. **The cost is small.** Self-host on Fly.io ($100/mo box) + Postgres free tier + Upstash already paid for in step 5. Amortized at 5% utilization across 30 tenants: $5/tenant/mo. At 1000 tenants: $9/tenant/mo (B1.6). Both inside the feasibility's 87-92% margin band.

**The new 4.9 spec text:**

> **4.9 Long-running tasks (hours).** Status: ✅. Source: D (split design).
>
> Jobs with expected duration ≤ 30 min run on **Vercel Workflows** with chained `sleep` and per-step retries. Jobs with expected duration > 30 min are dispatched by a router at job-start to **self-hosted Trigger.dev v4** on a single operator-owned project, with `payload.tenantId` identifying the tenant. Streaming progress for both paths is published to **Upstash Realtime** on `t:{tenantId}:ai-run:{runId}`. The buyer-facing surface (`useRealtime`, `useChat`) is identical regardless of runtime.

### B5.3 — Counter-argument 2: "30 min is plenty; chunk everything into sub-30-min jobs"

**The attack:** long jobs can be broken into sub-30-min chunks. AI agent runs: cap iteration count + checkpoint to Upstash + resume next invocation. ETL: chunk by row count. Cron: split across multiple invocations. Why is Trigger.dev still needed?

**Verifying chunking feasibility per workload shape:**

1. **AI agent runs (17.13):** Vercel AI SDK's `generateText` has a `stopWhen` parameter that calls `stepCountIs(N)` to cap iterations. With `maxSteps=25` (B2.1's recommendation), an agent doing tool-calling loops can be hard-capped. For the rare agent that needs >25 tool calls, the Function can checkpoint state to Upstash at step 25 (`SET agent:{runId}:state` with the conversation history), return 202 Accepted to the client, and a follow-up Function invocation resumes from the checkpoint. This is a manual pattern, not a queue primitive. **Cost:** engineering complexity ~2-3 days per AI surface. **Latency:** 5-10s of cold-start overhead per chunk. **Verdict:** feasible, but defeats the purpose of a streaming UX. The B1/B2 recommendation (Workflows `maxDuration=1800` for ≤30 min streaming) is better UX.

2. **ETL jobs (deferred to v2 per 17.20-17.23):** Trivially chunkable by row count. `SELECT * FROM raw_events WHERE id > last_id LIMIT 1000` is a 5-line pattern. **Not a v0 concern** because RAG is v2.

3. **Cron jobs (4.2, 4.4):** Already chunked by definition — a cron job is one short invocation per trigger. Vercel Cron + Workflows `sleep` covers this without chunking. **No Trigger.dev needed.**

4. **Multi-hour batch ingestion / data exports (v2):** Chunkable via Workflows `step.run()` calling itself in a loop with `sleep()` between iterations. Workflows retains state across pauses. **No Trigger.dev needed.**

5. **The single irreducible case — jobs that MUST process continuously for >30 min:** what would this look like in v0? Candidates:
   - Streaming a 4-hour response from OpenAI's o1-pro (does not exist; max thinking time is ~5 min).
   - Running a 2-hour video render on the operator's Vercel Function (out of scope for v1).
   - Mining a Bitcoin block in the operator's Function (out of scope).
   - **There is no real DeesseJS v0 job that MUST run continuously for >30 min.**

**Counter-counter argument:** "what about the buyer who needs to do bulk imports of 100k records?" — this is a one-shot bulk load, not a long-running job. It is a 5-15 min job that fits inside `maxDuration=1800` (Vercel Function with Fluid Compute + active-CPU billing). If the buyer needs to ingest 1M records, chunking at 100k/Function × 10 invocations is the right pattern, and Workflows handles it natively.

**Score on counter-argument 2:** the chunking argument is technically correct but irrelevant. The only DeesseJS workloads that exceed 30 min are AI agent runs and HITL pauses (which are paused, not running). For both, **Workflows is the correct primitive**, not chunked Functions. Chunking is the wrong abstraction for streaming agent UX. The counter-argument does not save Trigger.dev; it removes the only remaining excuse for it.

### B5.5 — Counter-argument 4: "Trigger.dev is a third-party primitive — lock-in risk"

**The attack:** Trigger.dev is a third-party vendor. If they change pricing, deprecate a feature, or go down, what is the migration cost? Vercel Functions are first-party (well, first-party to the platform the buyer chose).

**Vendor lock-in audit:**

| Vendor | Lock-in surface | Migration cost if vendor changes |
|---|---|---|
| **Vercel (Functions/Queues/Workflows/Cron)** | Every tenant is a Vercel project; the entire codebase deploys via Vercel. | High — migrating off Vercel means rewriting the runtime AND the queue AND the cron. But this is the **architectural lock-in the feasibility doc already accepts** ("the locked stack is Vercel-first"). |
| **Upstash (Realtime)** | Channels are Upstash Redis Streams. Migration target: Pusher, Ably, or self-hosted Redis Streams + SSE. | Medium — replaces the `useRealtime` hook calls (~30 sites) with the new SDK. ~2-3 days of code. |
| **Trigger.dev v4 Cloud** | Tasks live in `trigger.config.ts` + `trigger/` directory; deploys via Trigger.dev CLI; observability in Trigger.dev dashboard; secrets in Trigger.dev per-project env. | High — every `@trigger.dev/sdk` import becomes a Vercel Workflows `step.run()` or a Vercel Function call. The migration is the ~3-5 days B2.5 estimated, plus 1 week of QA. |
| **Self-hosted Trigger.dev v4** | Same code, but on the operator's infra. | Low — same Trigger.dev APIs, just relocated. Migrate by `docker compose pull` to new host. |
| **Turso** | libSQL protocol, SQLite dialect. Migration target: Neon (Postgres) or PlanetScale. | Medium — SQL dialect migration, schema translation. ~3-5 days. |
| **Resend** | React Email templates + Resend transport. | Low — swap transport, keep templates. ~1 day. |
| **Stripe** | Stripe Connect Standard — buyer's own connected account. | Low — operator has zero state to migrate. |

**Comparison: Trigger.dev Cloud vs Vercel Queues/Workflows:**

- **If Trigger.dev raises prices 2x:** v0 cost goes from $115-385/mo to $230-770/mo. v1 cost goes from $19,625/mo to $39,250/mo. The 30-200% cost increase is **not catastrophic** but is operator-painful. Migration path: 3-5 days code + 1 week QA. **Tolerable.**
- **If Trigger.dev deprecates a feature (e.g., `task.wait()`):** the migration is forced. Cost: same 3-5 days + 1 week QA. **Tolerable.**
- **If Trigger.dev goes down (acquisition, shutdown):** the migration is forced and urgent. Cost: same 3-5 days + 1 week QA. **Manageable if migration is pre-built.**
- **If Vercel changes Workflows pricing:** v0 cost goes from $30/mo (B1.6 estimate) to whatever Vercel sets. v1 cost goes from $1,500/mo to whatever. This is the same shape of risk, but the lock-in surface is larger (every tenant is a Vercel project, not a Trigger.dev project).

**Key insight:** both Trigger.dev Cloud and Vercel Queues/Workflows are third-party SaaS primitives with the same shape of vendor risk. The lock-in argument does not distinguish them. The argument only has force when comparing Vercel-native (larger lock-in surface but more strategic) vs Trigger.dev Cloud (smaller lock-in surface but less strategic). **If the operator is willing to accept Vercel lock-in for the runtime, they should accept Vercel lock-in for the queue** — the lock-in is the same vendor.

**Score on counter-argument 4:** attack fails. Trigger.dev Cloud is **not** meaningfully less lock-in than Vercel Queues/Workflows. Both are third-party SaaS primitives. The actual differentiator is per-tenant isolation (Vercel native wins by construction) and strategic alignment with the rest of the stack (Vercel native wins because the whole stack is Vercel). The lock-in argument favors Vercel-native.

### V1.1 — What "active CPU" actually means (the claim's biggest assumption)

Vercel's billing doc is vague. Direct fetch of `https://vercel.com/docs/functions/usage-and-pricing` (last_updated 2026-03-19) returns the **exact** line items:

> **Active CPU:** "This is the CPU time your code actively consumes in milliseconds. You are only billed during actual code execution and not during I/O operations (database queries, like AI model calls, etc.). Billed per CPU-hour. Pauses billing when your code is waiting for external services."

> **Provisioned Memory:** "Memory allocated to your function instances (in GB). Billed for the entire instance lifetime in GB-hours. Continues billing while handling requests, **even during I/O operations**. Each instance can handle multiple requests with optimized concurrency. **Memory is reserved for your function even when it's waiting for I/O**. Billing continues until the last in-flight request completes."

> "When the first request arrives, Vercel starts an instance with your configured memory. Vercel bills Provisioned Memory continuously until the last in-flight request finishes. **Vercel bills Active CPU only while your code is actually running. If the request is waiting on I/O, CPU billing pauses but memory billing continues.**"

What this means for an AI-heavy tenant:

1. **Active CPU is narrowly defined.** "Your code actively consumes" most plausibly refers to CPU time in V8 + user code execution. **Garbage collection IS user code execution** (V8 runs it on the same thread) — Vercel bills for it. **Event-loop ticks, Promise microtask resolution, JSON serialization of LLM stream chunks, AbortController cancellation** — all user code. They all bill.
2. **I/O wait is the only thing that doesn't bill.** The classic pattern `await llm.stream()` IS the only place the clock stops. Every other cost (function startup, environment validation, request parsing, response serialization, structured logging, telemetry export) bills.
3. **Provisioned Memory bills for the FULL instance lifetime including I/O waits.** This is the hidden cost the prior research under-weighted. A 1GB function with a 4s-active / 10s-alive streaming agent call in iad1 bills 1GB × 10s/3600 × $0.0106 = $0.0000294 per call. The Active CPU bills only 4s/3600 × $0.128 = $0.0001422. Memory is 21% of the per-call cost in this profile, not negligible.
4. **Cold start is explicitly NOT excluded from Active CPU.** V8 module load, package imports, Drizzle init, Better Auth instance creation — all bill. On a cold path, expect 200-500ms of pure active CPU before the first `await` even happens.
5. **What about `waitUntil`?** Per the Vercel Fluid Compute doc, `waitUntil` continues executing after the response is sent. This is the standard 4.9 pattern. **Active CPU during `waitUntil` is billable** because it's still "your code actively consuming CPU." Memory continues billing until `waitUntil` resolves. This is a NEW billing surface the pre-Fluid model didn't have.

**Net effect:** the prior research's "active CPU is nearly free for I/O-heavy work" is correct in concept, but the memory billing for the full instance lifetime, plus the active CPU during code execution around each I/O, plus the V8 GC pauses, plus the `waitUntil` post-response work, means the discount vs. wall-time billing is roughly 40-70%, not 90%+.

### V1.3 — The overage layer is dramatically underpriced

The feasibility's overage is $1.50/1M LLM tokens, $0.50/1M Vercel invocations, $0.10/GB Trigger.dev compute.

- **Vercel invocation overage at $0.50/1M**: the operator pays $0.60/1M for invocations + $0.128/CPU-hr + $0.0106/GB-hr for active CPU and memory. - **Trigger.dev compute overage at $0.10/GB**: this is correct (matches the published Trigger.dev pricing).

**The overage layer is not a margin rescue, it is a margin hole.** Every High-band AI-heavy tenant costs the operator MORE than the overage recovers. ### V1.4 — The 17.15 meter accuracy problem (B4's structural failure)

## Section A3 — Turso per-tenant DB at 1000+ scale

**Author:** A3 specialist. **Scope:** validate Turso (libSQL) as the per-tenant database for DeesseJS Cloud. Focus on the scale question: v0 = 30 tenants, v1 = 1000+ tenants. Document the group model, auth tokens, provisioning time, and compare to Neon / Convex / PlanetScale / Supabase on the dimensions that matter for a per-tenant SaaS.

### A3.1 — TL;DR

1. **Turso's "group" abstraction is per-region, not per-tenant** — it bundles one or more databases (DBs) with a primary location plus zero or more read-replica locations, all in the same logical group. For a 1000-tenant SaaS that wants a separate DB per tenant, the operator has two choices: (a) all 1000 DBs in one group (works on paper; the docs do not publish a per-group DB count limit), or (b) one group per tenant or per region (sharded, more operations surface, more cost).
2. **The Turso documentation does not publish a hard cap on databases per group, nor a per-group connection limit.** The `libsql` client uses HTTP, so "connection" is request-scoped, not socket-scoped — this is a different model from Neon/Postgres. The published limits are storage, rows, syncs, and PITR window per plan tier; database-count limits are listed only at the Free plan (100 DBs).
3. **The "4s to mint a Turso DB" figure in the feasibility is plausible** but not benchmarked. The platform API `POST /v1/organizations/{org}/databases` returns the `DbId` and `Hostname` synchronously, so it is in the same request — likely 1-3s in practice. At 1000 tenants/day, the bottleneck is operator-side parallelism, not the Turso API. **Turso does not publish per-org API rate limits**; the constraint is the 100 databases per org on the Free plan (forces the operator onto Developer $4.99/mo or higher by tenant ~10) and the 100-org-level project cap on the Free plan.
4. **Auth tokens are short-lived JWTs (default `never` expiry, configurable to `2w1d30m` format), with `full-access` or `read-only` scopes.** The blast radius of one leaked token is the single DB it was minted for — a strong isolation primitive compared to Neon's single-project-with-many-branches model. Rotation is per-DB (`POST /v1/.../databases/{name}/auth/rotate`) or per-group (`POST /v1/.../groups/{name}/auth/rotate`). There is no documented per-token revocation; rotation invalidates all tokens at once.
5. **For v0 (30 tenants): Turso is the right answer.** Cheap ($4.99/mo Developer plan covers all 30 with unlimited DBs), fast to provision, native Drizzle ORM support, libSQL is just SQLite, and the per-DB token model is the right isolation primitive. **For v1 (1000+ tenants): Turso is still viable, but the operator should test the per-group DB count threshold directly with Turso support** (feasibility Open Q #4) before committing to the single-group pattern. The 1-DB-per-tenant pattern with 1-group-per-region is a sane v1 architecture; the 1-group-per-tenant pattern is over-engineering.
6. **Neon, Convex, PlanetScale, and Supabase all have material trade-offs** for a per-tenant SaaS. **Neon** is the strongest alternative — Postgres-native, Drizzle-first-class, $0.106/CU-hr + $0.35/GB-mo on Launch, 100 projects per org on Free, 1,000 on Scale, autoscale-to-zero, branching. The downside: Neon "projects" are full Postgres databases with full CPU, so per-tenant cost is higher (~$10-30/tenant/mo at low band vs Turso's $0.10-1.00). **Convex** is per-developer pricing, not per-tenant-DB; the data model is document-oriented, not SQL; Drizzle is not applicable. **PlanetScale** is overkill and now expensive (Postgres plans from $15/mo; the Vitess plans from $39/mo are not what we need). **Supabase** brings auth + DB + storage but is the same per-project model as Neon with auth bundled; per-tenant cost is $25-50/mo at low band because each project has its own Postgres compute.

### A3.2 — Turso group model: what a group actually is

From the Turso Platform API documentation, a **group** is a logical container within an organization with these properties (`Group` schema, https://docs.turso.tech/api-reference/groups/retrieve):

| Field | Type | Description |
|---|---|---|
| `name` | string | Group name, unique across your organization |
| `uuid` | string | Universal unique identifier |
| `primary` | string | Primary location key (e.g., `us-east-1`) |
| `locations` | array (deprecated) | Location keys the group is located (now derived from the instance list) |
| `version` | string (deprecated) | libSQL server version |
| `delete_protection` | boolean | If true, group and DBs cannot be deleted |

A **group has one primary location and zero or more read-replica locations**. All databases within a group share the same primary. A database itself has multiple "instances" — one primary + N replicas — listed via `GET /v1/.../databases/{name}/instances`, with each instance carrying `uuid`, `name` (location code), `type` (`primary` or `replica`), `region`, `hostname` (https://docs.turso.tech/api-reference/databases/list-instances).

**This is per-region, not per-tenant.** The group abstraction groups databases that should be replicated together (same primary, same set of replicas). The Turso platform does not require that a group contain all the operator's databases. The operator can create one group per region and put all tenant DBs in each group, or one group per (region, plan) pair to keep pricing tiers isolated, or one group per tenant (overkill).

**The Turso documentation does not publish:**
- A maximum number of databases per group.
- A maximum number of connections per group.
- A maximum number of groups per organization.
- A per-org API rate limit for database creation.

The closest published limits are at the **plan level** (https://turso.tech/pricing):

| Plan | Price | DBs | Storage | Rows read | Rows written | Syncs | PITR |
|---|---|---|---|---|---|---|---|
| Free | $0 | 100 | 5 GB | 500M | 10M | 3 GB | 1 day |
| Developer | $4.99/mo | Unlimited | 9 GB + $0.75/GB | 2.5B + $1/B | 25M + $1/M | 10 GB + $0.35/GB | 10 days |
| Scaler | $24.92/mo | Unlimited | 24 GB + $0.50/GB | 100B + $0.80/B | 100M + $0.80/M | 24 GB + $0.25/GB | 30 days |
| Pro | $416.58/mo | Unlimited | 50 GB + $0.45/GB | 250B + $0.75/B | 250M + $0.75/M | 100 GB + $0.15/GB | 90 days |
| Enterprise | Custom | Custom | Custom | Custom | Custom | Custom | Custom |

**The Free plan's 100-DB cap is the only published DB-count limit.** The Developer, Scaler, and Pro plans all say "Unlimited databases." The 100-DB cap forces the operator onto Developer at tenant ~10 — a $4.99/mo cost that is noise in the COGS.

**Plan restriction on multi-group:** "Creating more than one group is limited to Scaler, Pro, and Enterprise plans" (https://docs.turso.tech/api-reference/groups/create). v0 with 30 tenants in a single group is fine on Developer. v1 with per-region groups (e.g., `us-east`, `eu-west`, `ap-southeast`) requires Scaler at minimum, which is $24.92/mo — still noise.

### A3.3 — At what tenant count do we migrate from 1 group to N groups?

The Turso group is per-region, so the right time to add a new group is **when the operator wants to add a new region**, not when the operator wants to add more tenants. Tenant count is a free axis within a region. **At 1000 tenants in one region, the pattern is "1 group per region, 1000 DBs in the group."**

When this pattern breaks down:

1. **Failure-domain isolation.** A Turso group has one primary instance; if the primary region has an outage, all 1000 DBs in the group become unavailable. Migrating to N groups lets the operator put the highest-tier tenants on a dedicated group with a higher SLA (Pro or Enterprise). This is a per-tier concern, not a per-region concern.
2. **PITR window differentiation.** PITR is per-organization, not per-group. The plan sets the PITR window (10/30/90 days). All DBs in all groups share the same plan. To give Enterprise tenants a 90-day PITR while Standard tenants have 10-day, the operator must run two organizations (one on Pro, one on Developer) — the operator already controls both.
3. **Per-tenant performance isolation.** libSQL groups share the same primary. A noisy tenant's workload affects all other tenants in the same primary region. Turso does not publish per-DB QPS or CPU isolation. This is the *only* reason to shard tenants into separate groups: per-tenant performance isolation for AI-heavy tenants.
4. **Org-level billing isolation.** All databases in an org share the same plan and the same overage. If the operator wants Standard and Enterprise tenants on separate bills, separate orgs.

**Recommendation for v0:** one group per region, all tenants in the same group. **Recommendation for v1 (1000 tenants):** same pattern, but with **a second organization on the Pro plan for the top 50 AI-heavy tenants** (per-tenant performance isolation, separate billing line, 90-day PITR). This is documented in the migration runbook, not in v0 code.

**The "1 group per tenant" pattern is wrong.** It adds 1000 group names to manage, 1000 sets of read replicas to provision (or not), and no isolation benefit over "1 group per region, all tenants in the group, but the operator's app code uses the per-DB auth token."

### A3.4 — Turso auth tokens: model, rotation, blast radius

Token API: `POST /v1/organizations/{org}/databases/{name}/auth/tokens` (https://docs.turso.tech/api-reference/databases/create-token).

| Token property | Value | Notes |
|---|---|---|
| **Format** | JWT (returned as a string in the response) | Returned in `{ "jwt": "..." }` |
| **Default expiry** | `never` | Configurable via `?expiration=` query param, e.g., `2w1d30m` |
| **Authorization scope** | `full-access` (default) or `read-only` | Set via `?authorization=` |
| **Permissions map** | `{ read_attach: { databases: [...] } }` | For ATTACH-database read permissions |
| **Token invalidation** | `POST /v1/.../databases/{name}/auth/rotate` — invalidates ALL tokens for a DB | Per-DB only; not per-token |
| **Group-level invalidation** | `POST /v1/.../groups/{name}/auth/rotate` — invalidates ALL tokens for all DBs in the group | Nuclear option |

**The blast radius of one leaked token is the single DB it was minted for.** This is the right isolation primitive for a per-tenant SaaS. If tenant-42's token leaks, the operator rotates tenant-42's token and the blast radius is contained to tenant-42.

**Comparison to other databases:**

- **Neon:** uses a single API key per project, with role-based access control inside Postgres (CREATE ROLE, GRANT). The blast radius of one leaked Neon key is the entire project. The operator would need one project per tenant to match Turso's isolation. **Turso's per-DB token model is more granular than Neon's per-project API key.**
- **Convex:** uses deploy keys per deployment; the entire project is one deployment, so the blast radius is the whole tenant. Same as Neon.
- **PlanetScale:** uses a single password per database; the blast radius is the DB, not the whole project. Similar to Turso.
- **Supabase:** uses two keys (anon, service_role) per project; the blast radius is the project. Worse than Turso for per-tenant isolation.

**Token rotation pattern for v0:**

1. The operator stores no raw tokens in the control plane DB — only the per-tenant `DEESSEJS_SECRET` JWT (B3 design). The `DEESSEJS_SECRET` carries the `turso_token` claim.
2. The `DEESSEJS_SECRET` has a 365-day expiry. The Turso token inside it can have a shorter expiry (e.g., 90 days) so the JWT can be re-signed without re-minting the Turso token.
3. When the operator wants to rotate a tenant's Turso token (e.g., leak response, scheduled rotation): call `POST /v1/.../databases/{name}/auth/rotate`, mint a new token, sign a new `DEESSEJS_SECRET` with the new Turso token claim, push the new `DEESSEJS_SECRET` to the Vercel project env via `PATCH /v9/projects/{id}` with new `environmentVariables[]`. 30-60s Vercel rebuild per rotation (per C1.4).
4. The boot handler `/api/internal/boot` re-validates the JWT, re-resolves the Turso token, and writes it to the per-process LRU cache + `/tmp/.deessejs-resolved.json`. All subsequent requests use the new token.

**Storage of the master Turso token (the one that mints per-tenant tokens):** keep it in the operator's Hono secret store (or a sealed Vercel KV key) — never in code, never logged. The feasibility doc's "no raw credentials stored in control plane DB" is correct.

### A3.5 — Provisioning time: 4s for 1000 tenants?

The feasibility doc's 4s figure for `POST /v1/organizations/{org}/databases` + token mint is **plausible but not benchmarked**. The platform API endpoint is synchronous: it returns the new DB's `DbId` and `Hostname` in the response. libSQL primary region selection + DB bootstrap + token mint = 1-3s in the same call. The second call (token mint) is sub-second.

**At 1000 tenants/day:** the bottleneck is **the operator's onboarding pipeline**, not the Turso API. Per C1.5, the four mint operations (Vercel, Turso, Upstash, Trigger.dev) run in parallel via `Promise.all`. The slowest is Vercel (~5s for the project create). Turso at 4s is in the pack. **Linear scaling holds at 1000 tenants/day** because the mint operations are parallelized per-tenant.

**Turso API rate limits (not published):** the Turso documentation does not document a per-org rate limit on the platform API. The constraint that matters is the **organization-level database count limit** (100 on Free, unlimited on Developer+). The control plane must track the running count and refuse to onboard if the org is at the cap.

**Per-region latency for the libSQL client:** the libSQL client connects over HTTPS. The `TURSO_DATABASE_URL` looks like `libsql://<db-name>-<org>.turso.io` or `https://<db-name>-<org>.turso.io`. From a Vercel Function in `iad1`, the round-trip to a Turso primary in `us-east-1` is 1-2ms. The libSQL client uses HTTP request/response, not a persistent connection (per https://docs.turso.tech/sdk/python/quickstart — "connects to your database via HTTP — no local file needed"). This means **"connection count" in the Postgres sense is not applicable**; the relevant per-tenant primitive is request rate, not connection count. Turso's per-database throughput is governed by storage tier and the underlying libSQL server, not by "max connections."

| **1000 tenants, low usage** (same per tenant) | $4.99 + ($0.75 × 970) = **$732.49/mo** (assumes avg 1 GB/tenant, 970 GB overage) | 1000 × 0.5 CU-hr × 720 hr × $0.106 = **$38,160/mo** + $0.35 × 1000 GB = $350 | $0.222/CU-hr × 180,000 CU-hr = **$39,960/mo** + $350 storage |
| **30 tenants, high usage** (10 GB DB, 2 CU avg, 50M row-reads, 5M row-writes/tenant/mo) | $4.99 + ($0.75 × 270) = **$207.49/mo** (270 GB over) | 30 × 2 CU × 720 × $0.106 = $4,579 + $0.35 × 300 = $105 = **$4,684/mo** | n/a |

**Neon is 10-20x more expensive than Turso at per-tenant SaaS scale.** The reason: Neon's compute is per-CU-hour at full price; Turso bills per-row-read and per-row-written, which is much cheaper at low-usage SaaS workloads (a SaaS with 1k MAU per tenant does ~1M queries/day, not 100k). Neon autoscale-to-zero helps, but the Launch plan requires a paid plan (Free tier is 100 CU-hr/project which a single Neon "project per tenant" would burn in 1-2 hours of active use).

**Per-tenant limits on Neon:**

- **Free plan:** 100 projects per org, 10 branches each. At 30 tenants = 30 projects + 30 main branches + ~50 ephemeral branches. **Fits.** 1000 tenants = 1000 projects. **Does not fit on Free** — must be on Launch or Scale.
- **Launch plan:** 100 projects per org. **At 30 tenants, fits.** At 1000, must use Scale.
- **Scale plan:** 1,000 projects per org (increasable). **Fits 1000 tenants exactly.**

**Regional support (Neon):** AWS regions (us-east-1, us-east-2, eu-central-1, ap-southeast-1, ap-southeast-2, etc.). Same EU-only deployment story as Turso.

**Per-tenant performance isolation:** Neon's autoscale + scale-to-zero means an idle tenant costs $0 compute. A spike scales up the CU on demand. **Stronger isolation than Turso at the per-DB level**, but at 10-20x the per-tenant cost at low band.

**Verdict on Neon:** stronger product (Postgres, branching, mature Drizzle story), **wrong cost shape for a per-tenant SaaS at 1000+ tenants**. A 1000-tenant Neon Launch deployment at low usage is $38,160/mo in compute alone — **doubles** the Studio tier's $999 sell price for the operator. The math only works at the AI-heavy / High band where each tenant is doing enough work to justify the compute.

**Hybrid pattern (worth considering):** Turso for Standard/Team/Studio tiers, Neon for Enterprise. Enterprise tenants have higher per-tenant usage, can absorb the per-CU cost, and benefit from Postgres-only features (PostGIS, pgvector, TimescaleDB, advanced indexing). This is the **only** pattern that makes economic sense.

### A3.7 — Alternative: Convex

Convex is a reactive document database, not SQL. Pricing (https://www.convex.dev/pricing):

| Plan | Price | Model | Free tier | Deployments |
|---|---|---|---|---|
| Free | $0 | 1-6 developers | 1M function calls, 0.5 GB DB, 1 GB files | 40 |
| Starter | PAYG | 1-6 developers | Same as Free with overage | 40 |
| Professional | $25/dev/mo | 1-20 developers | 1M function calls, 0.5 GB DB | 300 |
| Business & Enterprise | $2,500/mo min | 50+ developers | Custom | Unlimited |

**Per-tenant database in Convex:** Convex has one **deployment** per project, not per tenant. The "project" is the top-level billing unit. There is no per-tenant database primitive in Convex's model. A multi-tenant SaaS on Convex would use one project with `tenantId` columns on every table, **not** per-tenant deployments.

**Drizzle ORM with Convex:** **Not applicable.** Convex is its own data layer with its own TypeScript schema (`convex/schema.ts`). The DeesseJS 17.x surface is built around Drizzle on SQLite/Postgres. Migrating to Convex is a full rewrite of the ORM layer.

**Verdict on Convex:** **wrong product for DeesseJS Cloud.** The data model (document, not relational), the pricing model (per-developer, not per-tenant), the Drizzle incompatibility, and the lack of per-tenant isolation primitives all make this a non-starter.

### A3.8 — Alternative: PlanetScale

PlanetScale pivoted from MySQL/Vitess to Postgres in 2024. Current pricing (https://planetscale.com/pricing, AWS us-east-1 as of 2026-06-19):

- **Postgres EBS HA (3-node cluster):** $15/mo (PS-5) to $5,599/mo (PS-2560).
- **Postgres EBS non-HA (single-node):** $5/mo to $1,867/mo.
- **Postgres metal (3-node with local storage):** $50/mo to $94,429/mo.
- **Vitess non-metal (MySQL, 3-node):** $39/mo to $7,199/mo.

**Per-tenant DB in PlanetScale:** the PlanetScale equivalent is a "cluster" or a "branch" off a cluster. A 3-node cluster is $15-5,599/mo — a cluster per tenant is not economically viable. The branch model (one cluster, many branches) is the same as Neon's project model, but PlanetScale branches inherit the parent cluster's compute. A 1000-tenant SaaS would have 1 cluster ($15-50/mo) + 1000 branches (free or nominal cost).

**Drizzle ORM with PlanetScale:** first-class for Postgres. Drizzle Kit has `dialect: 'postgresql'` for PlanetScale.

**Per-tenant isolation:** a branch is a logical Postgres database in a shared cluster. The blast radius of one leaked branch credential is the branch, not the cluster. Similar to Neon's branch model.

**Verdict on PlanetScale:** **wrong cost shape for per-tenant SaaS.** 3-node cluster at $15/mo minimum is 2-3x more expensive than Neon Launch for the same workload. The branch isolation is similar to Neon. The only win is PlanetScale's "non-metal" 1-node option at $5/mo — but that's a single point of failure, which is a non-starter for a multi-tenant SaaS with a 99.9% SLA promise. **Skip PlanetScale.**

### A3.9 — Alternative: Supabase

Supabase is Postgres + auth + storage + edge functions in one product. Pricing (https://supabase.com/pricing):

| Plan | Price | DB Storage | Egress | Projects |
|---|---|---|---|---|
| Free | $0 | 500 MB | 5 GB | 2 active (paused after 1 week inactivity) |
| Pro | $25/mo | 8 GB (then $0.125/GB) | 250 GB | Unlimited |
| Team | $599/mo | 8 GB (then $0.125/GB) | 250 GB | Unlimited |
| Enterprise | Custom | Custom | Custom | Custom |

**Per-tenant DB in Supabase:** one project = one Postgres database = one auth instance = one storage bucket. A 1000-tenant SaaS would have 1000 projects on Pro (unlimited) at $25/mo minimum each = **$25,000/mo** in Pro subscription fees alone (before compute and storage). That's a hard no.

**Compute add-on:** each project runs its own compute instance (Micro to 16XL, $10-3,730/mo per project). For a 1000-tenant SaaS, even at the Micro $10/mo, the compute bill alone is $10,000/mo on top of the $25,000/mo subscription. **Total floor: $35,000/mo for 1000 tenants at zero usage.** Not viable.

**Drizzle ORM with Supabase:** first-class for Postgres. Supabase exposes a Postgres connection string; Drizzle connects via `node-postgres` or `postgres-js`.

**Per-tenant isolation:** same as Neon — one project per tenant. Auth is bundled, which is a win for the DeesseJS auth surface (17.7+).

**Verdict on Supabase:** **wrong cost shape for per-tenant SaaS.** Bundled auth is appealing, but the per-project compute subscription is prohibitive. A 1000-tenant Supabase Pro deployment is $35k/mo floor; the Turso equivalent is $730/mo. **Skip Supabase for the per-tenant data plane.** (Note: Supabase is already in the locked stack considerations for the auth layer, not the per-tenant DB. Reaffirm: the per-tenant DB stays on Turso or Neon; the auth layer is Better Auth on Drizzle, with Supabase not in the locked stack.)

### A3.11 — Migration triggers: when to revisit the per-tenant DB choice

The Turso 1-DB-per-tenant pattern is sound through 1000 tenants. Triggers to revisit:

1. **Tenant complains about Postgres-specific feature (pgvector at scale, PostGIS, TimescaleDB, full-text search with custom dictionaries, advanced JSON indexing).** Move that tenant to a Neon "Enterprise tier" dedicated project. v0 is the v0 design, not the v1 design.
2. **Tenant exceeds 50 GB / 250M row reads / 25M row writes on Turso Developer ($4.99 + overages) such that the per-tenant bill crosses $50/mo on Turso.** Move that tenant to Neon Launch where the compute is included in the price and the cost is more predictable. This is the high-end Studio / Enterprise band.
3. **Tenant requests EU-only data residency with BYOK encryption.** Turso Enterprise (custom) supports this; Neon Scale supports EU regions + IP allow + private networking. Either works.
4. **Operator hits a Turso org-level cap (e.g., 1000+ DBs in one org starts showing latency in the Turso dashboard or platform API).** Turso does not publish this cap, but the v1 architecture should plan for a sharding strategy: orgs by region (us-east, eu-west, ap-southeast) and within each org, groups by tier (standard, premium).
5. **Turso raises prices materially or ships a breaking change to the libSQL wire protocol.** Migration to Neon Launch becomes the default. Drizzle's dialect abstraction makes this a 1-2 day migration per tenant.

### A3.12 — Recommendations for v0 (A3-specific)

1. **Stay on Turso for the per-tenant DB.** Developer plan ($4.99/mo) covers all 30 v0 tenants with unlimited DBs. Single group per region, all tenants in the group. Document the migration trigger to a second organization on the Pro plan for AI-heavy tenants.
2. **Mint per-tenant auth tokens at provisioning time with 90-day expiry and `full-access` scope.** The token lives in the `DEESSEJS_SECRET` JWT, not in the control plane DB. The boot handler exchanges the JWT once and caches the resolved token in `/tmp/.deessejs-resolved.json`.
3. **Add a `read-only` auth token mint to the boot handler for the 11.1 notifications read path.** Read paths use a separate token so a leak of the read path does not give write access. The Turso API supports this via `?authorization=read-only` (https://docs.turso.tech/api-reference/databases/create-token).
4. **Add a `rotate-secret` endpoint to the operator admin dashboard.** Calls `POST /v1/.../databases/{name}/auth/rotate`, mints a new token, re-signs the `DEESSEJS_SECRET`, pushes to Vercel env. 30-60s rebuild per rotation. v0 cadence: 90-day rotation per tenant = ~10 rotations/month at 30 tenants, ~1/day.
5. **Document the "single group per region" pattern in the operator runbook.** Add a `migrate-tenant-to-premium-group` operation that creates a second group on the Pro plan, transfers the tenant's DB (`POST /v1/.../groups/{name}/transfer` — exists per the index at https://docs.turso.tech/llms.txt but not detailed in this research), and updates the `DEESSEJS_SECRET`. This is the AI-heavy-tenant escape hatch.
6. **Track per-tenant Turso usage via the `GET /v1/.../databases/{name}/usage` endpoint.** Returns `rows_read`, `rows_written`, `storage_bytes`, `bytes_synced` per instance and total. This is the input to the 17.15-style per-tenant meter and to the overage-pricing layer. Add a daily cron that pulls usage for all tenants and writes to the operator's Turso meter table.
7. **Add the `top_queries` endpoint to the operator observability stack.** `GET /v1/.../databases/{name}/stats` returns the top queries by rows read and rows written. The operator dashboard surfaces slow queries per tenant — useful for the C3 escape-hatch "eject to self-host" debugging flow.
8. **Test the 1-group / 30-DB pattern end-to-end with the first 3 design partners before onboarding tenant #4.** Verify: (a) provisioning time stays under 5s, (b) boot handler can reach the Turso DB from a Vercel Function in `iad1`, (c) `top_queries` returns useful data, (d) `usage` data is accurate enough to bill against. **This is the empirical test the feasibility doc's Open Q #4 needs.**
9. **Pre-negotiate a Turso Enterprise contract for the v1 (1000-tenant) scale.** The Enterprise plan supports BYOC, dedicated cloud, custom SLAs, and is the only path to a 99.95% SLA. v0 does not need Enterprise; v1 absolutely does.
10. **Test the libSQL embedded replicas pattern for low-latency reads.** A Vercel Function in `iad1` reading from a Turso primary in `us-east-1` is 1-2ms, which is fine. For Enterprise tenants on the EU tier, a Vercel Function in `fra1` reading from a Turso primary in `eu-west-1` is also 1-2ms. Embedded replicas (local file synced from the primary) are not needed for Vercel Functions (no persistent local file) but would matter if DeesseJS later ships a CLI or desktop app that needs offline reads.

### A3.13 — Open questions for the synthesis agent

1. **Per-group DB count threshold:** the Turso docs do not publish a max. **The v0 must benchmark this with the first 30 tenants and escalate to Turso support if any of the 30 onboardings take >10s or show dashboard latency.** A safe upper bound for "1 group per region" is probably 5,000-10,000 DBs, but this is engineering intuition, not a documented number.
2. **Turso Enterprise contract terms:** Pro plan is $416.58/mo. Enterprise is custom. **The synthesis agent should request a Turso Enterprise quote before v1 launch (1000+ tenants) to validate the unit economics.** Estimate: $1k-3k/mo for the 1000-tenant deployment with EU region + BYOK + 99.95% SLA.
3. **Per-tenant per-region replication:** can a tenant's DB be in multiple groups (one per region) and have writes replicated across? **The docs say "primary" is per-group; a single DB has one primary instance.** For a tenant who needs US + EU read replicas with a US primary, the architecture is: one DB in a us-east group (primary us-east, replicas in us-west, eu-west, ap-southeast). **The v0 design assumes us-east primary + us-west replica for Standard; eu-west primary for EU-only Enterprise.** This is sufficient for the buyer needs in the feasibility doc; multi-region active-active is not a v0 or v1 requirement.
4. **libSQL wire-protocol stability:** is libSQL on a long-term stable protocol, or does Turso reserve the right to break it on a major version? **The v1 architecture should plan for a 1-2 day Drizzle migration to Neon as a contingency.** Cost: 1-2 engineer-days per 100 tenants (re-mint auth tokens, re-point env vars, re-test boot handler). Not a v0 concern.
5. **Turso + Vercel integration:** the Turso docs list a Vercel integration (https://docs.turso.tech/integrations/vercel). **The v0 design should use this integration for the initial project setup, not for per-tenant provisioning (it mints env vars at the Vercel project level, not per-deployment).** Confirm: the integration is for the operator's bootstrap, not for the multi-tenant flow.
