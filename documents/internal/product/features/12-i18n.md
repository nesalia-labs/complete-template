# 12. i18n

> en only in v1. Architecture is translatable (routing, content, emails, dates, currency) so adding `fr` / `de` / `es` later is a docs step, not a build step. No full multi-locale UX in v1.

| # | Feature | Status | Source | Note |
|---|---|---|---|---|
| 12.1 | Architecture i18n-ready | ✅ | C | Routing, content, emails translatable |
| 12.2 | English only ships in v1 | ✅ | C | Confirmed decision |
| 12.3 | Routing translatable (path-based) | 🔵 | — | v2. Architecture reserved. |
| 12.4 | Content translatable (MDX) | 🔵 | — | v2. en only in v1. |
| 12.5 | Emails translatable | 🔵 | — | v2. en only in v1. |
| 12.6 | Date / currency / number formatting | ✅ | C | Intl APIs; en-US default |
| 12.7 | Time zone handling | ✅ | U | Stored as UTC, displayed per user TZ |
| 12.8 | RTL support (Arabic, Hebrew) | ⚪ | — | v2. Architecture supports; not built. |
| 12.9 | Locale auto-detection | ⚪ | — | v2. |
| 12.10 | Translation service integration (Crowdin, Lokalise) | ⚪ | — | v2. Buyer wires their own. |

**Notes:**
- i18n is the kind of thing that's expensive to retrofit and cheap to reserve. We reserve the architecture and ship `en`.
