# Delete tests

## What this is

A delete test verifies that a feature can be removed from the codebase without breaking anything else. The test:

1. Snapshots the current code state.
2. Removes the feature (deletes files, removes wiring, drops schema).
3. Verifies the app still works (types compile, tests pass, build succeeds).
4. Restores the state (atomic).

**If the test fails, the feature is "not deletable" — that's a P0 bug.** Per the "no half-built features" principle, every feature must be removable.

## The interface

```bash
pnpm test:delete --feature=<name>
```

Where `<name>` matches the feature module's directory name under `apps/template/packages/<name>/`.

The command:

- Runs `pnpm vitest run apps/template/packages/<name>/tests/delete.test.ts` (or equivalent).
- The test script orchestrates the snapshot/remove/verify/restore cycle.
- Exits 0 on success, non-zero on failure.

## The pipeline (step by step)

### Step 1 — Snapshot

The script creates a list of files to be touched (the feature directory + the 4 wiring points):

```
apps/template/packages/<feature>/
apps/template/packages/db/schema.ts                ← barrel export
apps/template/packages/api/root.ts                 ← contract registration
apps/template/env.ts                                ← env schema validation
apps/template/apps/web/app/(app)/layout.tsx        ← nav entry (if UI)
apps/template/content/docs/features/<feature>.md  ← doc page (if applicable)
```

These files are backed up to `/tmp/<feature>-delete-backup/` before any change.

### Step 2 — Remove

The script:

1. `rm -rf apps/template/packages/<feature>/` — deletes the entire feature directory.
2. Removes the barrel entry from `apps/template/packages/db/schema.ts` (one line: `export * from '../<feature>/db/schema';`).
3. Removes the router entry from `apps/template/packages/api/root.ts`.
4. Removes the env entry from `apps/template/env.ts`.
5. Removes the nav entry from the layout (if UI).

### Step 3 — Verify

The script runs:

```bash
pnpm tsc --noEmit          # types must compile
pnpm test:unit             # existing tests must pass
pnpm build                 # the app must build
```

If ANY step fails, the script stops, restores, and exits non-zero with the failing step's output.

### Step 4 — Restore (atomic)

The script restores the files from `/tmp/<feature>-delete-backup/` to their original locations. This happens whether the verification passed or failed (we always restore, so the developer can re-run the test or continue working).

After restoration, the working tree is back to the pre-test state.

### Step 5 — Report

The script prints a summary:

```
[delete:test] notes
  ✓ snapshot created
  ✓ feature directory removed
  ✓ barrel export removed
  ✓ router entry removed
  ✓ env entry removed
  ✓ nav entry removed (UI present)
  ✓ tsc passes
  ✓ unit tests pass
  ✓ build succeeds
  ✓ state restored
OK: notes is deletable
```

## What the test asserts

The delete test asserts the **modular contract**:

- The feature is self-contained in its directory.
- No other module imports from inside the feature (only from `index.ts`).
- Removing the feature doesn't break types, tests, or build.
- Removing the feature doesn't break the app's runtime (routes, pages, components).

## The special case: deleting `auth/`

The `auth/` module is special because it composes 8-10 Better Auth plugins. Deleting it is more complex.

### What the auth delete test does

The auth delete test:

1. **Snapshots** the same way as a normal feature.
2. **Detects all `apps/template/packages/auth/plugins/` files** to know which Better Auth plugins to uninstall.
3. **Detects all `apps/template/packages/auth/schemas/` files** to know which generated schemas to remove.
4. **Detects all references** to Better Auth's tables in:
   - `apps/template/packages/db/schema.ts`
   - `apps/template/env.ts` (env vars used by plugins)
   - `apps/template/packages/api/root.ts` (if Better Auth exposes any procedures through oRPC)
   - `apps/template/apps/web/app/(app)/layout.tsx` (the user menu)
5. **Removes** the `auth/` directory, all detected wiring, and the npm deps from the root `package.json`.
6. **Runs** `pnpm install` to uninstall the npm packages.
7. **Runs** `pnpm better-auth migrate --down` to roll back Better Auth migrations.
8. **Verifies** tsc, tests, build.
9. **Restores** everything (the npm install is part of restoration).

### Why the auth delete test is special

- Plugins are npm packages; removing them is not just deleting files.
- Schemas are generated by a CLI; removing them means re-running migrations down.
- The auth module is referenced from many places (org plugin, admin plugin, Stripe plugin, etc.).

### When to run the auth delete test

Less often than for normal features. The auth module is stable; it changes rarely. Run it:

- When upgrading Better Auth to a major version.
- When adding/removing plugins (e.g. enabling `sso` plugin).
- When changing the Stripe plugin configuration significantly.

## The special case: deleting a feature that uses Better Auth's tables

If a feature depends on Better Auth's `users`, `orgs`, `members`, etc. (e.g. our hypothetical `notes` references `orgId` and `authorId`), the delete test for that feature must:

- NOT remove Better Auth's tables (those are owned by `auth/`).
- Verify that Better Auth's tables still exist after deletion.

The `notes` example has `orgId: text('org_id').notNull()` — a foreign-key reference, but not a table-level dependency. The delete test for `notes` doesn't touch Better Auth's tables.

If a feature has a true FK constraint to a Better Auth table, the FK must be defined as a soft reference (string ID, not a DB-level FK). The auth delete test would then need to drop the FK constraint as part of restoration, then re-add it. This is rare and complex.

## CI integration

The delete tests run on every PR:

```yaml
# .github/workflows/ci.yml
jobs:
  delete-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - run: pnpm install
      - name: Delete test for each feature
        run: |
          for feature in $(ls apps/template/packages/); do
            if [ -f "apps/template/packages/$feature/tests/delete.test.ts" ]; then
              echo "Testing deletability of $feature"
              pnpm test:delete --feature=$feature
            fi
          done
```

The PR fails if any delete test fails.

## Failure modes and debugging

### "Type error after removing the feature"

Someone imports from inside the feature (e.g. `import { foo } from '@deessejs/notes/db/schema'` instead of `import { foo } from '@deessejs/notes'`).

**Fix:** update the import to go through `index.ts`. Then re-run the delete test.

### "Test fails after removing the feature"

A test in another module imports from inside the deleted feature.

**Fix:** update the test to import from the feature's `index.ts`. The test was violating the modular contract; the delete test caught it.

### "Build fails after removing the feature"

A Next.js page, layout, or component imports from inside the deleted feature.

**Fix:** update the import to go through `index.ts`. Or, if the page was specific to the feature, delete it too (the nav entry deletion should have handled this).

### "Schema migration conflict"

A migration in another feature references a column from the deleted feature's table.

**Fix:** the other feature should use soft references (string IDs), not hard FKs. If a hard FK exists, refactor to soft reference.

### "Restored state doesn't match snapshot"

This is a bug in the delete test script itself. File a bug against the test infrastructure.

## The buyer-facing link

The buyer-facing "Deleting features" doc (per `product/onboarding.md`) is a **human-rendered version of the same contract**. The buyer reads:

> To delete a feature: remove the directory, remove the imports, remove the schema, remove the env vars. The rest of the app continues to work.

The delete test **enforces** this promise in CI. If the contract breaks, the CI breaks.

When a feature changes in a way that makes it less deletable (e.g. someone adds a cross-feature import), the delete test catches it before merge.

## Implementation notes

The delete test script lives at `scripts/test-delete.sh` (or `scripts/test-delete.ts` if we use Node). It's invoked by both:

- The CI workflow above.
- The feature module's `tests/delete.test.ts` (which `exec`s the script).

The script is **self-contained**: no dependencies on the test framework. It can be run standalone for debugging.
