# i18n

## Decision

**English only for v1.** No i18n library ships in the template's `package.json` for v1. The **architecture is translatable** (routing shape, content shape, date/currency formatting, time zone handling) but no library is wired in.

When v2 adds locales (post-launch, demand-driven), the chosen library will be **`next-intl`** — but this is a v2 decision, not a v1 lock-in.

- **Decision date:** 2026-06-17
- **Decision owner:** tech lead
- **Status:** Locked (en only for v1)

## Scope (v1)

In v1, we ship:

- **Architecture i18n-ready** (feature 12.1). Routing shape, content shape, MDX, email templates, dates/currency/time zones all designed so adding locales is a docs step, not a build step.
- **English (`en`) only** (feature 12.2). The default locale.
- **Date / currency / number formatting** via native `Intl` APIs (feature 12.6). No extra dependency.
- **Time zone handling** — stored UTC, displayed per user TZ (feature 12.7).

We do **not** ship in v1:

- A routing-translatable structure (`/fr/...`, `/de/...`) — deferred to v2 (12.3).
- Content translation (MDX in multiple locales) — deferred to v2 (12.4).
- Email translation — deferred to v2 (12.5).
- RTL support (Arabic, Hebrew) — v2 (12.8).
- Locale auto-detection — v2 (12.9).
- Translation service integration (Crowdin, Lokalise) — v2, buyer-wired (12.10).

## Why "en only for v1" (not "i18n from day 1")

| Trade-off | i18n from day 1 | en only for v1 |
|---|---|---|
| Setup time | Higher (locale config, routing, content structure) | Lower (just `en`) |
| Maintenance cost | Higher (every string wrapped in t() even when only `en` exists) | Lower (no extra layer) |
| Buyer perception | "International-ready" feels premium | "English-only SaaS template" feels limited |
| Add locale later | Trivial | Easy if architecture is reserved |
| Lock-in to a library | High (must use the same library forever) | None (we pick when we actually need it) |

The "Apple" positioning argues for **finishing what we ship** rather than **shipping for every locale we might want**. v1 ships en, polished. v2 adds locales when there's buyer demand.

## Architecture reserved (the key)

Even though no library ships, the architecture is ready:

- **Routing**: `app/(marketing)/`, `app/(app)/`, `app/(system)/` — route groups are locale-ready (we can add `/[locale]/` later without restructuring).
- **Content**: `content/docs/en/`, `content/blog/en/` — future `content/docs/fr/`, etc., without restructuring.
- **Emails**: React Email templates accept a `locale` prop. The buyer passes `en` for now.
- **Dates**: `new Intl.DateTimeFormat(locale)` — locale-aware from day 1.
- **Currency**: `new Intl.NumberFormat(locale, { style: 'currency', currency })` — same.

When the v2 library is added, the existing code doesn't change. We add the locale route segment and the library handles the rest.

## v2 candidate: `next-intl`

When v2 adds locales, the candidate is **`next-intl`**.

| Library | Why it would win |
|---|---|
| **next-intl** (chosen as v2 candidate) | Built for Next.js App Router. Native i18n routing. Server Components compatible. ICU message format. Active development. |
| next-i18next | Older, less App Router-native. Less active. |
| react-i18next | React-only, doesn't help with Next.js routing. |
| lingui | Good but smaller ecosystem. |
| rosetta | Less mature. |
| FormatJS | Bigger, more opinionated. |

We don't lock `next-intl` now (no library is shipped). When v2 decides, we re-evaluate with current data.

## Buyer customization in v1

Even though the template ships `en`, the buyer can:

- Add their own locale support by following the architecture's i18n-ready shape.
- Use any i18n library they want — the architecture doesn't prevent it.
- Override the `Intl` defaults with their own date/currency formatting.

## Constraints

- **No i18n library in `package.json` for v1.** Adding `next-intl` or any other lib adds maintenance cost for a feature we don't use.
- **All user-facing strings inline in components** for v1. No `t('key')` wrapping.
- **Use `Intl` APIs for dates and currency.** No `date-fns`, no `moment`, no `dayjs`.
- **Routing shape stays locale-extensible.** Route groups, no hard-coded `/en/` paths.

## Failure modes

| Failure | Mitigation |
|---|---|
| Buyer wants to ship in French day 1 | The architecture is ready. They add `next-intl` themselves. |
| We need to add a locale for marketing (e.g. Spanish) | v2 decision. We re-evaluate `next-intl` and apply the architecture's reserved shape. |
| A buyer adds their own i18n lib that conflicts | Modular contract: they own their `apps/template/`. We can't prevent it. |
