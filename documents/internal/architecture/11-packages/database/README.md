# `packages/database`

The database package. Owns the Postgres schema, the typed data-access functions with mandatory `orgId`, the Drizzle client setup, and the migration workflow. Built on **Drizzle ORM 0.45.2** (the latest 0.x stable — see [`decisions/0002-drizzle-v1-rc.md`](./decisions/0002-drizzle-v1-rc.md) for why we pin to 0.x and the v1 migration plan).

## Purpose

Provide a single typed access layer for the rest of the system:

- **Schemas** (`pgTable` definitions) for every table, per feature module.
- **Zod schemas** auto-derived from `pgTable` via the `drizzle-zod` package (separate dep in 0.x; integrated into `drizzle-orm/zod` in v1).
- **Data-access functions** per feature module, each taking `(db, orgId, ...args)` explicitly.
- **Migrations** generated by `drizzle-kit`, committed, reversible.
- **Drizzle client** factory with the schema auto-discovery.

No design patterns, no factories, no classes — just plain functions. The `orgId` is always an explicit parameter, never hidden in a constructor.

The package has zero dependencies on `packages/api`. The dependency direction is `apps → packages`, never the reverse (per [`../10-decisions/0012-template-as-pattern.md`](../../10-decisions/0012-template-as-pattern.md)).

## Surface

Public exports from `packages/database/src/index.ts`:

| Export | Purpose |
|---|---|
| `Database` (type) | The Drizzle db instance type. |
| `createClient({ env? })` | The Drizzle client factory. |
| `loadEnv()` / `Env` | Zod-validated env schema. |
| `withSerializableRetry` | Postgres `40001` retry helper for `serializable` transactions. |
| `OrgId` / `UserId` | Branded ID types. |

Per-feature data-access **functions** are exported from modules like `@deessejs/database/users`, `@deessejs/database/projects`, etc. — one module per feature. Each function takes `(db, orgId, ...args)` and is fully typed.

`packages/api` consumes these functions directly. The schemas are internal but re-exported for oRPC input/output. The Drizzle client is created per-request by the auth-context middleware in `packages/api`.

**Note on relations.** In 0.x, relations are declared **per-table** via `relations(table, ...)` and exported alongside each `pgTable`. They are auto-discovered from the schema namespace (no separate `relations/` directory needed). See [`relations.md`](./relations.md).

## How `packages/api` consumes this package

The api package imports from `packages/database` through the data-access functions and the Zod schemas. It does **not** import `drizzle-orm` directly. Lint rules enforce this in `packages/api/src/core/**` and `packages/api/src/middleware/**`.

```
oRPC procedure (packages/api/src/core/<feature>/procedures.ts)
  → imports * as users from '@deessejs/database/users'
  → calls users.findById(ctx.db, ctx.orgId, input.id)
  → function takes the Drizzle instance and the orgId
  → function injects orgId into every query
  → Drizzle talks to Postgres
```

The procedure has no idea what Drizzle is. The data-access function has no idea what oRPC is. The schema has no idea what tenant context is. **Each layer knows one thing and knows it well.**

## The `ApiContext` extension (in `packages/api`)

```ts
// packages/api/src/orpc/context.ts
import type { Database } from '@deessejs/database'

export type ApiContext = AuthContext & {
  db: Database              // the Drizzle instance for this request
  orgId: string             // the active org (validated against session)
  // ... (requestId, ctxLog, scopes, etc.)
}
```

The middleware `auth-context.ts` in `packages/api` is responsible for:
1. Calling Better Auth to get the session.
2. Validating the active org from `X-Org-Id` header or cookie.
3. Creating the `db` instance via `createClient()`.
4. Resolving `orgId` from the session.
5. Returning the full `ApiContext`.

## Versions and dependencies

| Package | Version | Why |
|---|---|---|
| `drizzle-orm` | `^0.45.2` | Latest 0.x stable. The Better Auth Drizzle adapter does not yet support v1 (see [issue #6766](https://github.com/better-auth/better-auth/issues/6766)). See [`decisions/0002-drizzle-v1-rc.md`](./decisions/0002-drizzle-v1-rc.md) for the version strategy and the v1 migration plan. |
| `drizzle-kit` | `^0.30.1` | Matches drizzle-orm 0.45.x. |
| `drizzle-zod` | `^0.5.x` (add when needed) | Zod schema generation from pgTable. Separate package in 0.x (will be integrated as `drizzle-orm/zod` in v1). |
| `postgres` (postgres-js) | `^3.4.5` | Driver choice. Universal (Neon/Supabase/Railway). |
| `zod` | `^4.x` | Required by `drizzle-zod` and by oRPC. |

**Note on the v1 plan.** When Better Auth ships v1 adapter support, we bump to `drizzle-orm@1.0.0` and migrate the relations to `defineRelations(...)`. Estimated: 1 day of work. Tracked in ADR 0002.

## Conventions

- **`packages/api` does not import `drizzle-orm`.** Enforced by ESLint `no-restricted-imports` in `packages/api/src/core/**` and `packages/api/src/middleware/**`.
- **Procedures do not call `db.select()` directly.** They import the data-access functions from `@deessejs/database/<feature>`.
- **A data-access function that operates on a resource by ID returns `null` if the resource is not in the active org.** Never 404. The existence stays opaque to non-members.
- **Migrations are committed SQL, generated by `drizzle-kit`.** Never hand-written, never edited post-generation without a "regenerate" PR.
- **Schemas live in `packages/database/src/schema/<feature>/`.** One folder per feature module, exported from `index.ts` (the modular contract).
- **Zod schemas are auto-derived** via `drizzle-zod` (added when the first feature needs them). Do not hand-write `createSelectSchema`/`createInsertSchema`/`createUpdateSchema` outputs.
- **Relations are per-table.** Each `pgTable(...)` file exports a `relations(table, ...)` alongside it. The schema namespace auto-discovers both.
