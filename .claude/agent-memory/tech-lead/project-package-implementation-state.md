As of 2026-06-22 (last re-checked during mail research), the DeesseJS template `apps/template` is a nested pnpm workspace with monorepo structure. Implementation state has progressed since the 2026-06-19 snapshot.

**State by package:**

- **`packages/database`** — M0 done. Schema (Better Auth tables, generated by CLI into `src/schema/auth/index.ts`), Drizzle client (postgres-js + drizzle-orm 0.45.2), env loader (`zod` schema, lazy `loadEnv()`), transactions helper (`withSerializableRetry` for SQLSTATE 40001), branded ID types (`OrgId`, `UserId`), PGlite-based test infra with cross-tenant WHERE-clause pattern test. NO feature modules (`src/users.ts`, etc. — M1+). NO per-table relations yet.
- **`packages/auth`** — M0 mostly done. Better Auth 1.6.19 configured with 8 plugins (email+password, emailVerification, magicLink, emailOTP, socialProviders Google+GitHub, organization, admin, apiKey, twoFactor, passkey). RBAC matrix (owner/admin/member/billing + project + billing statements). PGlite test instance, integration tests for factories/sessions/cookies. GAPS: `experimental.joins: true` not enabled, no `accountLinking` config, no `captcha` plugin, no `hooks/` directory, **`secondaryStorage` is a no-op placeholder** (rate limit auth broken in prod), **all 4 mail sends are `console.log`** (magicLink/OTP/passwordReset/emailVerification) — this is the bridge to [[project-mail-stack-decision]]. CLI pinned to `@better-auth/cli@1.4.21` while runtime is `1.6.19` (mismatch).
- **`packages/api`** — Sprint 1 done, NOT empty as the 2026-06-19 snapshot claimed. ~1500 LoC of working code: Hono app composition (`src/hono/app.ts`), 3 mounts (`/api/auth/*`, `/rpc/*`, `/api/v1/*`), full middleware chain (`requestIdMiddleware` → `authContextMiddleware` → `rateLimitMiddleware` (stub) → `logger()`), oRPC `publicProcedure` / `sessionOnly` / `authorized` builders, custom `WireError` serializer, AsyncLocalStorage `requestContext` for oRPC error encoder, RPCHandler + OpenAPIHandler + Scalar UI, single Better Auth org cast localized in `auth-client.ts`, 4 integration tests (smoke + failure-paths). 3 placeholder procedures (`ping`, `pingAuth`, `pingOrg`). NO feature business procedures. `rateLimitMiddleware` is a STUB — see [[project-rate-limit-stub]].
- **`packages/cache`** — Sprint 1 done. `createRedisClient()` factory with auto-detect (Upstash if env set, else Memory), `MemoryRedisClient` (Upstash-compatible in-memory backend with TTL/scan/keys), 9 typed key builders (auth + api + org-scoped), 19 unit tests.
- **`packages/mail`** — ✅ **IMPLEMENTED** (as of 2026-06-23, was empty in the 2026-06-22 snapshot). Full `sendMail()` API, 3 mailers (ConsoleMailer for dev, NoopMailer for tests, ResendMailer for prod) with auto-detect via `MAIL_PROVIDER` env, 7 React Email 6.6 templates (ResetPassword, VerifyEmail, MagicLink, OtpSignIn, OtpEmailVerification, OtpForgotPassword, Wrapper), resend@6.14. **Gap**: ConsoleMailer logs `react: '<ReactNode>'` placeholder — the verification/reset URL inside the React tree is not extractable from logs, breaking end-to-end dev testing of verify-email and reset-password flows. Decision pending: enhance ConsoleMailer to also dump rendered text, or build a dev mail inspector page. See [[project-mail-stack-decision]] for the stack rationale.
- **`packages/{admin, ai, i18n, logs, notifications, sdk, storage, ui, utils}`** — all EMPTY.
- **`apps/web`** — Next.js 16.2.9 + React 19.2.4 + Tailwind 4 + Geist fonts. **Only the create-next-app starter page** — landing page default, no product UI. No `src/app/api/[[...route]]/route.ts` Hono mount yet. ⚠️ AGENTS.md warns: "This is NOT the Next.js you know" — read `node_modules/next/dist/docs/` before writing any Next code (Next.js 16 breaking changes).
- **`apps/cli`** — EMPTY. Listed in pnpm-workspace.yaml.
- **`apps/docs`** — EMPTY. Listed in pnpm-workspace.yaml.

**Doc-vs-code inconsistencies observed (still):**
- Drizzle version: `schema.md` and `transactions.md` say `1.0.0-rc.3`; `README.md`, `relations.md`, `queries.md`, `tenant-isolation.md`, ADR 0002, and `package.json` all say `^0.45.2`. **ADR + code win**, the two outliers are stale.
- The architecture docs (`docs/internal/architecture/11-packages/*`) describe feature modules (`core/<feature>/procedures.ts`), a `registry.ts` pattern, etc. — none of this exists yet (M1+).
- The api package's `publicProcedure` is currently just an alias for `base` (line 10 of `public.ts`) — docs treat it as a distinct thing. OK to leave until first feature module needs the distinction.

**Why:** the architecture docs were written as a complete vision before code; implementation is catching up M0 by M0. As of 2026-06-22, packages/api is the surprise — it's ahead of what the 2026-06-19 memory captured.

**How to apply:**
- When reviewing or extending code, treat the architecture docs as the *target*, not the *current state*. Flag the gap explicitly to the founder rather than silently aligning code to a doc — many gaps are intentional ("M1+").
- When the user asks about packages/api: it IS implemented at Sprint 1 level (Hono + oRPC + 3 transports + auth context). Don't suggest scaffolding it from scratch.
- When the user asks about packages/mail: propose [[project-packages-mail-architecture]] as the plan; don't re-research from zero.
- Before any meaningful M1 work, identify which empty packages become blockers (mail is the immediate one — see [[project-mail-stack-decision]]).
